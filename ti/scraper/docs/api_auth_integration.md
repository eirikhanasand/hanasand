# API Auth Integration Notes The scraper API assumes authentication and authorization are enforced by the main CTI application or an API gateway before traffic reaches this service. ## Boundary - All external routes are versioned under `/v1`. - Forward `x-tenant-id` and `x-actor-id` from the trusted upstream boundary. - Forward request context headers when present: `x-request-id`, `x-trace-id`, `x-client-id`, `x-service-token-context`, `x-analyst-role`, and `x-source-approval-role`. - Resolve roles/scopes upstream; this service should receive already-authorized operational calls. - Require `source:write` or `scraper:admin` for source administration. - Require `intel:run` for `/v1/intel/plan` and `/v1/intel/runs`. - The scraper contract labels analyst roles, source approval roles, read scopes, and write scopes, but the CTI gateway remains the authorization decision point. - Audit records use tenant/requester/request/trace/principal identifiers and decisions only. They must not include bearer material, cookies, authorization headers, passwords, or secret values. ## Tenant Boundary - Tenant filtering must happen before pagination, DTO projection, evidence lookup, graph export, source administration, claim ledger access, restricted metadata review, capture listing, and run status lookup. - Cross-tenant misses should return the stable not-found path rather than revealing resource existence. - Local development may fall back to the single-tenant fixture mode; production must require `x-tenant-id`. ## Idempotency - Clients should send `idempotency-key` on `POST /v1/intel/runs`. - Idempotency is scoped by tenant ID. - Repeated requests return the original run instead of creating duplicate work. ## Pagination - List endpoints accept `limit` and `cursor`. - `nextCursor` is an opaque offset for the current in-memory implementation and should be treated as a string by clients. ## Rate Limits And Safe Errors - Gateway rate-limit responses should include `retry-after`, `x-rate-limit-policy`, and `x-request-id`. - `/api/ti/search` and `/v1/intel/search` are public-wrapper/live-search sensitive and advertise short retry delays so clients can keep the last safe answer visible while polling. - Source administration, public-channel apply plans, restricted metadata apply plans, and graph/STIX export routes advertise slower retry delays and remain dry-run or fail-closed when policy or queue pressure requires it. - Stable error bodies use `{ error: { code, message, details } }` with compact identifiers and retry hints only. Safe examples in `/v1/contracts.enterpriseApiSurface.errors.safeExamples` cover `queue_pressure`, `policy_blocked`, and `idempotency_conflict`. ## Gateway Cutover - `/v1/contracts.apiGatewayIntegration` is the gateway deployment plan for auth headers, public wrapper routing, rate-limit headers, tenant/requester propagation, cutover stages, and rollback triggers. - The gateway owns authentication and authorization, maps public `POST /api/ti/search` to canonical `/v1/intel/search` semantics, forwards only trusted identity/request context, and keeps token material outside scraper DTOs and logs. - Cutover stages are `shadow_headers`, `public_wrapper_shadow`, `canary_public_wrapper`, and `versioned_api_clients`; any missing identity, tenant-boundary failure, public POST failure, rate-limit header drift, error-envelope drift, raw-material leak, or SDK fixture drift should roll back to the compatibility wrapper and hold client promotion. - Run `bun run check:api-gateway` with the contract index, route inventory, API regression sentinel, SDK fixture check, full tests, and public scraper-native proof matrix before promoting gateway traffic. ## Public Wrapper Cutover Watch - `/v1/contracts.publicWrapperCutoverReadiness` is the final readiness/deprecation watch for moving public `/ti` traffic away from temporary outer-wrapper assumptions while keeping `POST /api/ti/search` compatible. - The public wrapper, scraper-native `/v1/intel/search`, run polling, SDK fixtures, API regression sentinel, gateway plan, and future SSE/webhook contract must agree on `status`, `runId`, cursors, `refreshAfterSeconds`, `updated`, `warningCodes`, `publicTiAnswer`, and `publicWrapperDelta`. - Unknown or unsupported queries may say `Searching`, `queued`, or `partial` with source gaps. They must not become ready from a default actor, stale cache, demo copy, or implicit APT29 example. - `POST /api/ti/search` remains an alias until the public proof matrix for `APT29`, `Random Actor`, and `Made Up Actor` is green, route/gateway/API regression checks pass, and the release board promotes scraper-native replacement. Roll back on default actor detection, demo copy, missing cursors, unstable run ids, tenant-boundary drift, rate-limit header loss, error-envelope drift, unsafe payload leaks, or streaming-contract drift. ## Frontend Progressive Updates - `/v1/contracts.frontendProgressiveUpdateContract` freezes the `/ti` UI polling and merge contract for first responses, repeated polls, empty deltas, new deltas, metadata review, graph holds, claim-ledger holds, no-result/searching states, and ready states. - `/v1/contracts.scraperNativeReplacementReadiness` publishes the release/readiness board for replacing fallback behavior with scraper-native search across frontend `/ti`, public `POST /api/ti/search`, and `GET /v1/intel/search`. - `/v1/contracts.darkwebIndexFrontendContract` freezes the `/ti/darkweb/index` metadata-only frontend contract: search, cursor pagination, category/risk/liveness/review filters, table columns, compact triage copy, provenance display, safe detail drawer, and "what was not accessed" text. - `/v1/contracts.sourceAtlasFrontendContract` freezes the `/ti/sources/atlas` dry-run frontend contract: source-candidate search, query-class/family/parser/legal/activation filters, table columns, safe detail drawer, first-100/first-1000/future-10k plan labels, export manifest review fields, approval packets, rollback packets, and canary-plan handoff copy. - `/ti/darkweb/index` clients should read `/v1/darkweb/status`, `/v1/darkweb/search`, and `/v1/contracts`; display only redacted labels, hashes, safe summaries, triage labels, liveness, provenance, review state, and blocked/held copy; and treat legal/risk labels as triage labels rather than legal advice. - `/ti/sources/atlas` clients should read `/v1/sources/atlas`, `/v1/sources/atlas/export`, `/v1/analyst/source-activation-packets`, and `/v1/contracts`; display domains, hashes, source families, query-class coverage, parser state, legal/robots state, source value score, approval state, rollback ids, and dry-run canary/export plans only. - Frontend clients merge by `runId` and `deltaCursor`, preserve the last safe `publicTiAnswer` on empty deltas or retryable degradation, append warning codes without duplication, and never backfill default actor or demo copy. - Unknown or made-up actor queries render compact `Searching` copy and continue polling; they must not show seeded actor context, stale cache prose, or generic clear-web facts as a ready answer. - UI DTOs expose route fields, cursors, warning codes, source coverage, review states, graph/claim caveats, and compact summaries only. They must never expose raw bodies, restricted raw URLs, credentials, authorization/cookie material, object keys, leaked rows, or webhook secrets. - Dark-web index UI DTOs must never expose raw unsafe URLs, full onion/I2P locators, credentials, raw bodies, object keys, leaked rows, private-message material, payload downloads, or threat-actor interaction content. - Source-atlas UI DTOs must never imply registry mutation, source-pack import, crawl enqueue, silent activation, private/invite/auth/CAPTCHA activation, credentialed fetch, or payload download. The atlas is a review/export/canary planning surface until a separate explicit approval workflow acts. - Replacement promotion requires local contract gates, full Bun tests, public proofs for `APT29`, `Random Actor`, and `Made Up Actor`, and board coverage for CVE/advisory, sector/country, victim/company, restricted metadata holds, graph holds, and empty-delta behavior. ## Realtime Prototype - `/v1/contracts.realtimeDeliveryPrototype` is a disabled-by-default SSE/webhook prototype contract. It is not a mounted delivery surface and must stay off unless a future release board explicitly enables the feature flags. - `/v1/contracts.realtimeDeliverySoak` is the disabled realtime soak gate for fixture-only SSE replay, webhook outbox retry, cursor gaps, fallback-to-polling, and no-leak event envelopes. - Required disabled flags are `TI_REALTIME_SSE_ENABLED=false`, `TI_REALTIME_WEBHOOKS_ENABLED=false`, `TI_REALTIME_DELIVERY_WRITES_ENABLED=false`, and `TI_REALTIME_ROUTES_ENABLED=false`. - Prototype events cover run status, answer deltas, evidence promotion, source gaps, graph holds, restricted holds, quality caveats, retry hints, and terminal states. Every event envelope carries only ids, cursors, hashes, warning/reason codes, confidence, and compact summaries. - Polling remains primary. Realtime disablement, delivery failure, replay gaps, subscription conflicts, or policy blocks must fall back to the same `runId`, `pollCursor`, `deltaCursor`, status, retry headers, and warning codes used by 3-second polling clients. - Realtime payloads must preserve public wrapper guardrails: no default actor, no demo or stale-cache prose, unknown queries say `Searching`, stable run ids/cursors, and no raw/unsafe payload fields. - Soak promotion remains `hold_realtime_polling_primary` until feature flags, mounted routes, outbox writes, cursor-gap behavior, no-leak serialization, public-wrapper field parity, and unknown-query semantics all pass a future release board. ## Client Generation Freeze - `/v1/contracts.clientGenerationFreeze` is the source of truth for generated TypeScript clients and future inert realtime types. It is contract-only and does not commit generated artifacts in the scraper repo. - Code generation must read `/v1/contracts` only. It must not inspect memory store rows, adapter internals, evidence objects, object storage keys, raw payloads, or restricted source locators. - Frozen targets are `typescript_fetch_browser`, `typescript_node_service`, `analyst_automation_types`, and `future_realtime_types`. The realtime target is disabled until feature flags are explicitly enabled; polling clients remain complete and primary. - Generation must fail closed on operation id drift, required schema loss, fixture drift, public wrapper field drift, cursor field drift, error envelope drift, tenant header drift, realtime flags enabled by default, or unsafe payload fields. - Required proofs before publishing generated clients are `bun run check`, `bun run check:api-regression`, `bun run check:sdk-fixtures`, `bun run check:route-inventory`, `bun run check:contract-index`, focused API/API-regression tests, and full `bun test`. ## Generated SDK Changelog Gate - `/v1/contracts.clientGenerationFreeze.changelogGate` is the generated-client release gate. It keeps BA.5 contract-only: the scraper publishes fixtures, changelog policy, and proof requirements, but does not commit generated SDK artifacts. - Major changes are blocked for generated clients when operation ids, required schemas, required fields, cursors, error envelopes, tenant/requester headers, disabled realtime defaults, or no-leak DTO boundaries would drift. - Minor changes may add optional fields, route metadata, SDK fixtures, inert realtime types, warning codes, or deprecation notices. Patch changes are limited to docs, examples, fixture timestamp refreshes without field drift, and release-board notes. - Deprecations require the published deprecation/sunset/link/version/request headers, at least 90 days notice, release-board approval, and continued `POST /api/ti/search` compatibility until scraper-native public proof is green. - The SDK fixture gate must include every `/v1/contracts.sdkIntegration.fixturePack.requiredFiles` entry and keep invariant fields plus no-leak assertions aligned before generated client release. ## Current Typed Routes - `GET /v1/health` - `GET /v1/metrics` - `GET /v1/sources` - `POST /v1/sources` - `PATCH /v1/sources/:id` - `GET /v1/frontier` - `GET /v1/intel/plans` - `POST /v1/intel/plan` - `POST /v1/intel/runs` - `GET /v1/intel/runs/:id` - `GET /v1/intel/runs/:id/results` - `GET /v1/captures` - `GET /v1/incidents` - `GET /v1/auth/integration-notes` ## Versioning And Deprecation - `/v1` is the current external API prefix. Additive changes may add optional fields, warning codes, schemas, examples, and route metadata without a new major prefix. - Removing required response fields, changing the shared error envelope, changing idempotency behavior, renaming cursor fields, changing public wrapper stable fields, or changing tenant/requester header names requires a new major route prefix. - Deprecation responses should use `deprecation`, `sunset`, `link`, `x-api-version`, and `x-request-id` headers with at least 90 days notice. - `POST /api/ti/search` remains a compatibility wrapper for public clients and maps to canonical `/v1/intel/search` semantics. It should not be sunset until public wrapper proof and the client compatibility matrix are green. - Deprecation examples in `/v1/contracts.enterpriseApiSurface.versioning.examples` must use only route names, request ids, replacement routes, and compact error envelopes. ## OpenAPI Contract Freeze - `GET /v1/contracts` is the source of truth for the OpenAPI-ready contract. - The `enterpriseApiSurface` section defines trusted gateway auth, tenant/requester identity, idempotency, pagination, rate-limit hints, audit fields, error envelopes, OpenAPI paths, schema components, and no-leak examples. - The `publicWrapperResponsiveAudit` and `publicWrapperDeltaAudit` sections freeze `/api/ti/search` compatibility for first responses, repeated polls, empty deltas, new deltas, queue pressure, policy blocks, searching/no-result responses, metadata review holds, graph holds, claim-ledger holds, and ready states. - The `clientCompatibilityMatrix` section freezes client expectations for frontend `/ti`, CTI backend integrations, analyst automation, future SDKs, and future SSE/webhooks. - The `apiRegressionSentinel` section freezes the backward-compatibility gate for stable route signatures, top-level contract keys, public search fields, OpenAPI schemas, auth/idempotency/pagination/error/rate-limit invariants, SDK fixture files, and no-leak assertions. Run `bun run check:api-regression` before SDK/client/public-wrapper promotion. - The `apiGatewayIntegration` section freezes gateway cutover behavior for forwarded identity, service context, route scopes, public wrapper mapping, rate-limit hints, rollout stages, and rollback-safe actions. - The `publicWrapperCutoverReadiness` section freezes the public wrapper cutover/deprecation watch, banned fallback assumptions, unknown-query copy rule, proof commands, rollback triggers, and Agent 02/06/07/08/10 handoffs. - The `realtimeDeliveryPrototype` section freezes the disabled-by-default realtime prototype, feature flags, event envelope, event prototypes, idempotency/replay semantics, fallback-to-polling behavior, no-leak payload rules, and Agent 02/06/07/08/10 handoffs. - The `realtimeDeliverySoak` section freezes disabled delivery soak scenarios, webhook outbox retry states, cursor-gap fallback actions, polling compatibility, release blockers, rollback actions, and no-leak event payload boundaries. - The `clientGenerationFreeze` section freezes generated-client targets, operation ids, schema manifests, fixture manifests, fail-closed drift checks, no-leak DTO rules, and release-board handoffs for codegen. - The `frontendProgressiveUpdateContract` section freezes `/ti` UI state mapping, polling merge semantics, empty-delta behavior, held-state rendering, proof scenarios, and no-leak UI payload rules. - The `scraperNativeReplacementReadiness` section freezes the scraper-native replacement decision board, proof matrix, blockers, rollback actions, public proof commands, and Agent 02/06/07/08/10 product handoffs. - The `darkwebIndexFrontendContract` section freezes `/ti/darkweb/index` as an SDK/OpenAPI-ready metadata-only surface with table filters, category/risk chips, safe detail drawer sections, source provenance, review states, cursor pagination, compact copy rules, and no-leak UI fields. - The `sourceAtlasFrontendContract` section freezes `/ti/sources/atlas` as an SDK/OpenAPI-ready dry-run source discovery surface with table filters, source-family chips, parser/legal/activation state, first-100/first-1000/future-10k plans, export/review packets, rollback packets, canary handoff fields, compact copy rules, and no-mutation/no-crawl UI boundaries. - The `streamingWebhookCompatibility` section freezes future SSE/webhook behavior as contract-only delivery. Polling remains primary and compatible: clients keep the same `runId`, `pollCursor`, `deltaCursor`, status values, retry headers, tenant/requester headers, and no-leak DTO boundaries whether they later subscribe to events or keep 3-second polling. - Required fields may be extended with optional fields, but removing or renaming required response keys, states, cursor fields, retry headers, or schema components requires a new compatibility entry before promotion. ## Client Compatibility Matrix - Frontend `/ti` uses `POST /api/ti/search` and `GET /v1/intel/search`, preserves `runId`, `pollCursor`, `deltaCursor`, `updated`, `publicTiAnswer`, and `publicWrapperDelta`, and keeps the last safe answer on empty deltas or retryable degradation. - CTI backend clients use idempotent `/v1/intel/runs`, run status, paginated results, and STIX export routes. They must honor `idempotency-key`, `nextCursor`, and the shared error envelope. - Analyst automation uses analyst loop, metadata review, claim ledger, and restricted metadata status routes. These responses remain metadata-only and victim-safe. - Future SDKs should generate from the `SdkPollingEnvelope` and `SdkSubscriptionRegistration` schemas plus the `xSdkPollingContract` extension. - Generated clients should also consume `clientGenerationFreeze` to lock operation ids, required schemas, fixture files, drift checks, and disabled realtime type boundaries before publish. - SDK compatibility CI should use `/v1/contracts.sdkIntegration.fixturePack` and `/v1/contracts.sdkIntegration.compatibilityCi` as the contract source for polling fixtures, empty deltas, queue pressure, policy blocks, idempotent reuse, cursor result pages, metadata review, future event payloads, and no-leak assertions. - Generated SDKs should not ship unless contract index, route inventory, `bun run check:api-regression`, `bun run check:api-gateway`, `bun run check:sdk-fixtures`, focused API tests, full Bun tests, and the public scraper-native proof matrix are green. - Future SSE/webhooks are contract-only: event payloads may carry identifiers, cursors, warning codes, hashes, confidence, and compact summaries, but never raw bodies, credentials, restricted raw URLs, object references, or leaked rows. - Event types cover run status, answer deltas, evidence promotion, source coverage gaps, graph review holds, restricted metadata holds, retry hints, and terminal states. Webhook failure handling must retry with bounded backoff, disable unsafe or failing subscriptions, and fall back to cursor polling without source activation or data expansion. - Prototype realtime client work should consume only the `RealtimeDeliveryPrototype` and `RealtimePrototypeEvent` schemas until mounted routes are explicitly approved. Generated clients must treat all realtime modes as optional and keep polling implementations complete.
