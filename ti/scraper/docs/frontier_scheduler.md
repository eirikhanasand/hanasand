# Frontier Scheduler Design ## Goal The frontier is the queue control plane between intelligence plans and adapters. It owns task ordering, leases, retry/backoff, fairness, and crawl budgets. Adapters only execute leased tasks and acknowledge the result. ## Embedded MVP - Keep the in-memory frontier for unit tests and local development. - Workers call `next()` to lease one eligible task. - Workers acknowledge with `complete`, `fail`, or `cancel`. - Expired leases are returned by `requeueExpiredLeases`. - Budgets are enforced before leasing by task count, reserved bytes, and wall-clock deadline. - Fairness keys cover tenant, intelligence request, source, adapter type, and optional custom keys so large actor sweeps do not starve routine feed collection. ## Postgres-Backed Queue First Use Postgres before Redis/NATS because it keeps queue state, run state, policy decisions, and audit reconstruction in one transactional system. Suggested tables: - `frontier_tasks`: task id, tenant id, run id, request id, source id, source type, target URL, priority, score breakdown JSON, queued/available/deadline timestamps, retry count, max retries, max bytes, budget key, fairness key, status. - `frontier_leases`: task id, worker id, leased at, lease expires at, attempt deadline, acknowledgement status, acknowledgement reason. - `crawl_budgets`: budget key, tenant id, request id, task limit, tasks leased, byte limit, bytes reserved, deadline. - `frontier_events`: append-only audit for enqueue, lease, complete, fail, cancel, lease expiry, budget rejection, and policy rejection. Lease query pattern: 1. Requeue expired leases in the same transaction. 2. Merge duplicate queued work by tenant, active-run reuse key, source, and target URL. 3. Select eligible tasks with `FOR UPDATE SKIP LOCKED`. 4. Exclude exhausted budgets, source backoff windows, per-source concurrency, and expired deadlines. 5. Order by effective priority plus work-class boost, age boost, and recent tenant/request/source penalties. 6. Insert/update lease row, increment budget counters, and write a `frontier_events` lease record. Acknowledgement pattern: - `complete`: mark task complete, close lease, emit event. - `fail`: close lease; if retry budget remains, set `available_at` with exponential backoff and source backoff; otherwise mark failed. - `cancel`: mark queued or leased task cancelled and close any lease. - `requeueExpiredLeases`: mark lease expired, decrement running counters through derived state, and return task to queued with `available_at = now`. ## Redis/NATS Later Use Redis/NATS only when Postgres queue contention or cross-region fanout becomes a measured problem. - Redis can cache ready-task buckets and source backoff windows, but Postgres remains the audit source of truth. - NATS can broadcast task availability and worker heartbeats, but acknowledgements must still be durably written. - Temporal is an option for long-running workflow orchestration, not for first-pass adapter fetch leases. ## API Coordination Agent 09 can expose: - `/v1/frontier`: paged queue plus grouped summary by tenant, source, adapter type, priority bucket, age bucket, and budget state. - `/v1/intel/runs/:id`: run status plus frontier context showing queued/leased counts and the same compact summary. - `/v1/frontier.scheduler`: production-cutover design, load fixtures, starvation alerts, active-run garbage-collection decisions, and per-task diagnostics. Diagnostics include work class, run reuse key, pressure state, queue age, fairness group, duplicate merge count, and next actionable time. - `/v1/frontier/apply-plan`: dry-run-only scheduler repair contract for Agent 09 and Agent 10. Responses include compact apply-plan DTOs, OpenAPI-ready examples, preconditions, expected queue/run deltas, rollback notes, approval level, risk class, API warning codes, emergency-brake headroom state, execution preview, and promotion packet link fields. Mounted endpoint proof: run `bun run check:frontier-apply-plan`. Expected output is compact JSON with `ok: true` and four scenarios: normal HTTP 200 with no actions, degraded HTTP 200 with `pause_noisy_source_queues`, overloaded HTTP 200 with `trigger_emergency_brake`, and invalid action HTTP 400 with `invalid_action`. Every scenario reports queue, lease, and run snapshots unchanged. Do not expose raw sensitive task payloads in metrics or logs. ## Production Cutover Contract The scheduler may move from embedded memory to Postgres or an external queue only if the planner contract stays stable: - `CollectionTask`, `CollectionRun`, planner decisions, and live-search DTO fields remain backend-neutral. - Active-run reuse keys are indexed durably so repeated `/ti` polling attaches to active work. - Acknowledgements are idempotent and append audit events before externally visible state changes. - Fairness decisions use the same work classes as the in-memory scheduler: public live search, analyst deep dive, source-health probe, restricted metadata sweep, background sweep, and replay/retention. - Stale active runs, abandoned polling clients, completed runs with pending deltas, and retryable failed runs are GC decisions first; mutation belongs to the storage/worker layer. - Apply-plan creation never leases tasks, acknowledges tasks, changes runs, starts workers, or mutates source state. The frozen action set is expired lease release, abandoned run cancel, transient requeue, low-priority deferral, noisy source pause, quarantine recommendation, and emergency brake. ## Load Model Fixtures The production-readiness fixtures cover 1, 10, 100, and 1,000 concurrent live-search pollers. Each fixture includes expected polls per minute, expected new run creation after reuse, analyst deep dives, source-health probes, restricted metadata sweeps, replay/retention jobs, and per-class queue-age thresholds. The intent is to prove live public polling creates attachment pressure, not duplicate run pressure.
