# Public Telegram Collection ## Scope - Collect only publicly visible Telegram channels that have been reviewed and approved as medium-risk sources. - Use official Telegram boundaries only: Bot API where it fits the source model, or an approved MTProto client library with documented API credentials. - Normalize every message to `CollectedItem` with channel, message id, message URL, collection time, published time, content hash, extractor version, and confidence in metadata provenance. ## Source Configuration Telegram sources use `SourceRecord` with: - `type`: `telegram_public` - `accessMethod`: `official_api` - `risk`: `medium` - `status`: `active` - `url`: public channel target such as `https://t.me/example_channel`, `tg://resolve?domain=example_channel`, or `@example_channel` - `legalNotes`: why public collection is allowed and how the source was reviewed - `approvedAt` and `approvedBy`: required before collection - `metadata.telegramApi`: `bot_api` or `mtproto_library` - `metadata.pageSize`: 1-100 messages per request, default 50 - `metadata.minIntervalSeconds`: per-source collection interval, default 60 - `metadata.afterMessageId` / `metadata.beforeMessageId`: pagination cursor state - `metadata.channelId` / `metadata.approvalScope`: optional public-channel identity and reviewed scope. - `metadata.topicTags`, `metadata.actors`, `metadata.ransomware`, `metadata.cves`, `metadata.victims`, `metadata.sectors`, and `metadata.countries`: safe focus metadata used by query-window planning. - `metadata.rateLimitResetAt`: scheduler-facing flood-wait/backoff state. - `metadata.minimizePii`: defaults to true - Prohibited metadata keys include `accountAutomation`, `autoJoin`, `joinGroups`, `joinChannels`, `privateChannel`, `inviteLink`, `sessionString`, `userSession`, `phoneNumber`, `password`, and `bypassAccessControls`. ## Query-Window Planning - Public Telegram planning ranks only approved `telegram_public` sources whose safe focus metadata matches actor aliases, ransomware families, CVEs, victim names, sectors, countries, or broad CTI topic tags. - `APT29` expands to common public aliases such as Cozy Bear, The Dukes, and Nobelium for channel selection. - Cybercrime aliases such as Scattered Spider, Octo Tempest, UNC3944, and Scatter Swine are expanded for channel selection. - Ransomware brand searches expand common family/rebrand terms for Akira, LockBit, BlackCat/ALPHV, and Cl0p/Clop while still remaining metadata-only until a public channel is approved. - CVE, victim, and sector query hints are normalized into query terms so source focus can match actor names, aliases, malware names, CVEs, victim names, sectors, and ransomware brands without broad private-channel discovery. - Rate-limited channels can still produce delayed tasks with `availableAt` set from `metadata.rateLimitResetAt`. - Private targets and account-automation configs are reported as blocked, not converted to tasks. ## Source Activation Bridge - `bridgeTelegramPublicActivationSource` converts Agent 01 public-channel source records into safe Telegram planning inputs by deriving topic, actor, alias, sector, and country focus from source catalog metadata. - The bridge strips private/account fields such as session strings, phone numbers, invite markers, automation flags, and media-download settings before returning planning metadata. - `buildTelegramPublicSourcePackCompatibility` reports Agent 01 source-pack compatibility for registry and candidate channels: coverage tags, approval state, channel publicness proof, retention class, and abuse-control defaults. - Compatibility DTOs redact prohibited private/invite targets to a blocked placeholder instead of returning the original unsafe Telegram URL. - `explainTelegramPublicCoverageGaps` classifies public-channel gaps for `/v1/intel/search`: no matching approved channels, matching channels pending review, disabled by policy, delayed by rate limits, or already queued. - `/v1/intel/search` exposes these public-channel `coverageGaps` alongside activation recommendations so the UI can explain whether the next step is approval, activation, waiting, or policy repair. - `buildTelegramPublicActivationProgram` groups Agent 09 outputs into recommended public packs, matching active channels, pending-review channels, rate-limited channels, disabled-by-policy channels, and no-approved-channel gaps. - `buildTelegramPublicReconciliation` compares source packs, registry approval, channel health, cursor state, rate limits, scheduler state, and evidence freshness into API-ready diagnostics. - Reconciliation diagnostics use stable statuses: active, approved idle, pending review, rate-limited, unavailable, policy-disabled, stale cursor, high duplicate URL rate, high edit/delete churn, and no query coverage. - Repair recommendations are deterministic and limited to safe operator actions: reduce window, delay poll, refresh cursor, quarantine channel, request review, activate source pack, and suppress repeated URLs. - `buildTelegramPublicCutoverReport` combines source-pack recommendations, reconciliation, cursor/rate-limit state, abuse controls, evidence freshness, and repairs for scraper-native search cutover rehearsals. - Cutover summaries expose Agent 09 fields for ready channels, pending review channels, rate-limited channels, stale cursors, high duplicate URL channels, safe partial evidence, and a deterministic next action. - `buildTelegramPublicApplyPlan` converts cutover repairs into dry-run operator steps only. Each step records prerequisites, scheduler/evidence effects, rollback, rate-limit safety, media metadata-only policy, PII minimization, and whether it is automation-safe, human-approval-required, blocked, or rollback-only. - Apply plans never activate private, invite, internal `t.me/c`, or account-automation targets. Unsafe public-channel records can only produce blocked review steps or quarantine-style remediation. - `buildTelegramPublicCanaryRollout` converts approved public sources and dry-run source-pack candidates into a rollout packet for first-channel and five-channel canaries. It captures source selection, bounded window sizing, rate-limit envelopes, query dedupe, language coverage, spam/churn detection, edit/delete replay, unavailable-window handling, Agent 06 evidence handoff verification, Agent 07 caveats, and Agent 10 release-train status. - `buildTelegramPublicPromotionCanaryProof` turns the rollout packet into a replayable promotion proof. It shows which public messages become Agent 06 evidence ledger rows, claim candidates, graph hints, source-health updates, Agent 07 public-answer caveats, and Agent 10 RC-gate decisions. - Promotion canary `sourceHealth` stays compact and safe: rate-limit debt, duplicate URL pressure, edit/delete churn, unavailable windows, language drift, spam/churn, evidence yield, claim yield, and rollback triggers. It preserves message URL/id/hash provenance without raw Telegram payloads, credentials, invite links, or account/session material. - `buildTelegramPublicPromotionCertification` converts the canary proof into the release-candidate certification packet. It declares the exact conditions under which public-channel evidence can influence public answers, Agent 08 graph hints, source-health telemetry, and Agent 10 release decisions. Available, replayable, promoted, high-confidence messages may influence answers and graph candidates; weak co-mentions, edited messages, repeated URLs, rate limits, unavailable/deleted posts, low-yield channels, language drift, and quarantine triggers are held, caveated, or blocked while still updating metadata-only source health. - `telegramPublicApplyPlanApiContract` freezes the standalone `/v1/public-channels/apply-plan` contract for Agent 09. The response is dry-run only and includes schema-ready examples for automation-safe, human-approval-required, blocked private target, rate-limited channel, and rollback-only quarantine cases. - The apply-plan API contract forbids raw message bodies, media payloads, credentials, session strings, phone numbers, passwords, invite links, and user-session material. - `/v1/public-channels/apply-plan` is wired through dedicated route helpers. Requests may optionally filter by known action names; unknown actions return `invalid_action` without producing an apply plan. - `/v1/public-channels/status` returns the production public-channel status DTO for one query. It combines queued task count, safe partial evidence, incremental poll buckets, promotion readiness, source-pack/cutover context, and the same safe-output guarantees without mutating sources, scheduler, or captures. - `/v1/public-channels/status` also exposes operator states for actively collectable, delayed, pending-review, quarantined, and policy-blocked channels plus source-pack compatibility, compact answer-readiness summaries, and operator-control answer-quality effects. ## Mounted Endpoint Proof - Start the scraper API locally with `bun run start`, then POST to `/v1/public-channels/apply-plan`. The endpoint is dry-run only and does not mutate source, scheduler, or evidence state. - Activation proof: `curl -sS -X POST http://127.0.0.1:8097/v1/public-channels/apply-plan -H 'content-type: application/json' -d '{"query":"Volt Typhoon","entityType":"actor","actions":["activate_source_pack"]}'` Expected fragment: `action:"activate_source_pack"`, `execution:"human_approval_required"`, `mode:"dry_run"`. - Human-review proof: `curl -sS -X POST http://127.0.0.1:8097/v1/public-channels/apply-plan -H 'content-type: application/json' -d '{"query":"APT29","entityType":"actor","actions":["request_review"]}'` Expected fragment: `action:"request_review"` with `execution:"human_approval_required"` for reviewable public sources. - Rate-limit proof: `curl -sS -X POST http://127.0.0.1:8097/v1/public-channels/apply-plan -H 'content-type: application/json' -d '{"query":"APT29","entityType":"actor","actions":["delay_poll"]}'` Expected fragment: `action:"delay_poll"` and `execution:"automation_safe"` when a matching source has rate-limit state. - Rollback-only quarantine proof: `curl -sS -X POST http://127.0.0.1:8097/v1/public-channels/apply-plan -H 'content-type: application/json' -d '{"query":"APT29","entityType":"actor","actions":["quarantine_channel"]}'` Expected fragment: `action:"quarantine_channel"` and `execution:"rollback_only"` for active unsafe channels. - Invalid-action proof: `curl -sS -X POST http://127.0.0.1:8097/v1/public-channels/apply-plan -H 'content-type: application/json' -d '{"query":"APT29","actions":["join_private_group"]}'` Expected fragment: `error.code:"invalid_action"` and no `applyPlan`. - Blocked-private-target proof: seed or configure a public-channel source with a private/invite/account-automation target, then request `request_review`. Expected fragment: `execution:"blocked"` and prerequisites include `blocked: private, invite, or account-automation targets cannot be activated`. - Safe-output proof: responses must not contain raw message bodies, media payloads, private metadata values, credentials, session strings, passwords, invite links, or user-session material. - Status proof: `curl -sS 'http://127.0.0.1:8097/v1/public-channels/status?q=APT29&entityType=actor&cursor=0'` Expected fragment: `endpoint:"/v1/public-channels/status"`, `safeOutput.rawMediaPayloadsExposed:false`, `poll.media.rawFetchAllowed:false`, and a status of `queued`, `partial`, `blocked`, or `ready`. ## Source Packs - Public Telegram source packs live as dry-run fixtures, for example `seeds/public_telegram_channel_packs.json`. Importing or validating a pack must not make network requests. - Packs are disabled by default and each entry starts as a candidate requiring operator review before activation. - Each entry must include public channel URL, topic coverage, actor/ransomware/CVE/victim/sector/country focus, legal notes, legal basis, license/compliance notes, retention class `public_chat_text`, approval scope, and bounded rate-limit expectations. - Validation rejects invite/private `t.me/+`, `joinchat`, and `t.me/c/` links; account automation/session fields; raw media download settings; missing legal/compliance metadata; and unbounded page or interval settings. - `buildTelegramPublicSourcePackReadiness` turns approved registry sources and dry-run pack candidates into the compact production-readiness packet: approval scope/state, collection window, rate-limit budget, duplicate URL policy, edit/delete handling, language and coverage hints, Agent 06 replayable ledger handoff, Agent 07 caveats, and Agent 10 release gate. - `/v1/intel/search` can receive public Telegram source packs from the API host and returns `publicChannel.sourcePackRecommendations` plus activation recommendations for exact pack entries that match random actor, ransomware, CVE, victim, and sector queries. - Takedown/legal concerns belong in `compliance.takedownContact`, legal notes, and review tickets. If a channel owner requests removal or terms change, disable the source and preserve only prior public evidence metadata required for audit. ## Live Search Backfill - `/v1/intel/search` can include a `publicChannel` section with `partial`, `pending_channel_search`, or `blocked` status. - Partial evidence DTOs expose only safe public-channel fields: source id, channel, message URL, message timestamp, snippet, extracted URLs, public forward metadata, reply id, language hint, media metadata-only summary, extraction handoff markers, message id/state, content hash, and confidence. - When no approved public channel can be searched, the response returns activation recommendations instead of pretending there are no public-channel results. - Recommendations identify matching candidate channels and the needed action: review, approve, activate, or fix policy. - These DTOs are designed for the outer `/ti` experience: show partial public evidence immediately, then queue or recommend safe public-channel activation for deeper refresh. ## Promotion And Polling - `promotePublicChannelEvidence` converts a partial public-channel DTO into a `CollectedItem` suitable for capture and extraction only when the message URL is a stable public `t.me/{channel}/{messageId}` URL that matches the evidence channel and message id. - Promotion preserves message-level provenance: source id/type, channel, message id, message URL, published time, promoted time, content hash, adapter version, confidence, and promoted evidence stage. - `buildTelegramPublicEvidencePromotionProgram` summarizes production promotion readiness for actor search. It separates promotable candidates, duplicate URL/hash suppression, edited messages, deleted/unavailable updates, blocked sources, policy-disabled sources, and official API rate-limit backoff. - `buildTelegramPublicRuntimeCollection` is the scheduler/evidence runtime boundary after an approved public-channel adapter run. It returns cursor windows, source patch metadata, scheduler acknowledgement hints, safe evidence DTOs, poll deltas, promotion handoff items, extraction inputs, duplicate suppression, edit/delete churn, rate-limit retry hints, and policy-disabled/unavailable states. - Runtime collection includes `connector`, an operator-ready contract for cursor leases, rate-limit state, bounded window sizing, source-health patches, public-message provenance, edit/delete/unavailable deltas, Agent 06 promotion handoff, and Agent 07 actor-readiness downgrade signals. - Runtime connector `answerReadiness` maps each public-message provenance row to a deterministic Agent 06 ledger id, delta kind (`new`, `edited`, `deleted_or_unavailable`, or `unchanged`), promotion status, and Agent 07 claim readiness (`ready`, `partial_evidence`, or `needs_review`) with caveat codes for edited, unavailable, rate-limited, partial, or repeated-URL evidence. - Runtime statuses are deterministic: `ready`, `partial`, `blocked`, `rate_limited`, `policy_disabled`, `unavailable`, `high_duplicate`, and `high_churn`. Scheduler code should acknowledge `ready`/`partial`/high-volume completion, retry `rate_limited` and transient `unavailable`, block policy-disabled sources, and quarantine non-promotable blocked runtime results. - Deleted or unavailable messages stay polling-visible as status updates but are not promoted into capture input. - `buildTelegramPublicIncrementalPollDto` returns new messages since a cursor, edited messages, deleted/unavailable statuses, promoted extraction ids, and the next cursor. - Incremental polling also exposes forwarded messages, URL-mentioned messages, explicit cursor state, and a media guarantee showing raw fetches are disabled. - Public-channel promotion stores media as metadata-only and rejects raw media payloads or source configs that request media downloads. - Metadata-only sensitive captures may retain safe entity hints such as victim and sector labels for profile display, but raw source text is removed before storage and never returned by public-channel status or search DTOs. - `buildTelegramPublicSourceHealthUpdate` summarizes per-channel provenance and health updates: last seen message id/date, fetch outcome, rate-limit reset, duplicate URL rate, deleted/unavailable rate, policy-block rate, and promoted message ids. ## Abuse Controls - `applyTelegramPublicAbuseControls` clamps repeated `/ti` public-channel windows to a per-channel limit from `metadata.publicQueryWindowLimit` or a conservative default. - Active source rate-limit state is reused as a denial/delay signal instead of issuing another public-channel request. - Channel cooldowns from `metadata.channelCooldownUntil`, requested language hints, topic coverage, and expanded query terms can block or delay repeated public-channel searches before collection. - Repeated public-channel URLs already seen by the caller are suppressed in the abuse-control DTO. - Raw media fetches are always disabled; media-heavy posts remain text plus safe media metadata only. ## Reliability Scoring - `buildTelegramPublicReliabilityReport` emits safe per-channel reliability DTOs for `/v1/intel/search` and `/v1/public-channels/status`. - Reliability metrics are bounded scores or ratios for freshness, duplicate URL pressure, edit/delete churn, unavailable windows, language coverage, topic fit, rate-limit/failure history, and promotion yield. - Ratings are `healthy`, `watch`, `degraded`, `quarantine`, or `blocked`. Degraded, quarantine, blocked, low-yield, stale, or high-churn sources are marked `needsReview` and `partialEvidenceOnly` where appropriate. - Repeated actor-query evidence and previously seen public message URLs produce `suppress_repeated_urls`; stale cursors produce `refresh_cursor`; rate limits produce `delay_poll`; edit/delete bursts produce `reduce_window` and review; unavailable or unsafe channels produce quarantine. - Non-English evidence is preserved when topic fit is strong. A language mismatch lowers language coverage but does not discard safe, relevant public evidence. - `buildTelegramPublicActorReadinessDto` maps reliability into Agent 07 readiness: noisy, unstable, unavailable, blocked, quarantined, or low-yield public-channel evidence stays `partial` or `needs_review` instead of becoming a ready actor answer. - `buildTelegramPublicCompactSearchSummary` keeps `/v1/intel/search.publicChannel.summary` small for the outer CTI app: latest public-channel evidence freshness, aggregate reliability, promotion-yield rating, operator-state counts, and answer-readiness status. - `/v1/intel/search.publicChannel.sourcePackReadiness`, `/v1/intel/search.sla.publicChannel.sourcePackReadiness`, and `/v1/public-channels/status.sourcePackReadiness` expose source-pack readiness without raw messages. Agent 01 can use coverage tags and pending candidates for coverage gaps, Agent 06 receives replayable ledger ids/message URLs, Agent 07 receives caveat codes, and Agent 10 receives per-source release gate status. - `/v1/intel/search.publicChannel.canaryRollout`, `/v1/public-channels/status.canaryRollout`, and `/v1/public-channels/apply-plan.canaryRollout` expose the dry-run canary state. Selected sources are approved public channels only; pending pack entries remain review candidates. The contract index advertises the canary routes and fields for integration. - `/v1/intel/search.publicChannel.promotionCanary`, `/v1/public-channels/status.promotionCanary`, `/v1/public-channels/apply-plan.promotionCanary`, `/v1/intel/search.sla.publicChannel.promotionCanary`, `/v1/intel/search.publicTiAnswer.publicChannelPromotion`, and `/v1/contracts.semantics.publicChannelPromotionCanary` expose the Task Y promotion proof. Consumers should read these fields for replayable evidence counts, claim/graph hint counts, rollback triggers, answer caveats, and source-health status instead of inferring readiness from raw public-channel messages. - `/v1/intel/search.publicChannel.promotionCertification`, `/v1/public-channels/status.promotionCertification`, `/v1/public-channels/apply-plan.promotionCertification`, `/v1/intel/search.sla.publicChannel.promotionCertification`, `/v1/intel/search.publicTiAnswer.publicChannelCertification`, and `/v1/contracts.semantics.publicChannelPromotionCertification` expose the Task Z certification packet. The packet includes decision rules, evidence certification, claim certification, graph certification, Agent 06 evidence certification, Agent 07 answer state-machine transition, Agent 08 graph certification, and Agent 10 RC decision/rollback actions. - `buildTelegramPublicSlaReport` emits `/v1/intel/search.publicChannel.sla` and `/v1/public-channels/status.sla` for Agent 10 release gates. It rolls up cursor freshness, collection success, rate-limit debt, duplicate URL pressure, edit/delete churn, unavailable windows, language/topic fit, promotion yield, ledger-backed claim yield, and answer-readiness impact into `pass`, `warning`, or `blocker`. - The SLA `releaseGate` block is owned by Agent 04 and exposes the Agent 10 runtime proof name `public_channel_sla`, proof command, rollback path, warnings, blockers, and release-decision impact without raw messages, private channel data, media payloads, credentials, or session material. - The SLA `enforcement` block converts each metric into explicit release checks. Stale cursors, rate-limit debt, low collection success, duplicate URL pressure, edit/delete churn, unavailable windows, low promotion yield, weak ledger-backed claims, and answer-readiness downgrades become `pass`, `warning`, or `hold` states with answer impact. - Enforcement also mirrors the Agent 06 ledger handoff state, Agent 07 answer-readiness state, and Agent 10 release-packet state so consumers do not infer readiness from raw public-channel evidence. - `buildTelegramPublicOperatorControlEffects` explains how dry-run controls affect answer quality: delayed polling keeps claims partial, reduced windows improve precision, quarantine stops untrusted evidence, review requests queue readiness, cursor refresh restores delta continuity, and repeated-URL suppression reduces duplicate claim pressure. - Reliability DTOs expose no raw Telegram message bodies, private channel data, credentials, session material, or raw media payloads. ## Pagination And Rate Limits - Adapter requests carry `afterMessageId` and `beforeMessageId` so clients can fetch forward or back through public history without guessing state. - Client responses may return `nextPagination` and `rateLimitResetAt`; those are preserved in run metadata and item metadata for scheduler/state storage. - Crawl-state output includes `channel`, current `afterMessageId` / `beforeMessageId`, next cursor ids, `rateLimitResetAt`, `lastMessageDate`, `lastMessageId`, and `fetchDurationMs`. - Scheduler ownership remains outside the adapter. The adapter reports source rate-limit hints but does not sleep or retry indefinitely. ## Failure Categories - `policy_blocked`: source policy, private/invite target, or prohibited account-automation config blocked collection before API use. - `rate_limited`: official API/library reported a flood wait or retry-after condition. - `timeout`: official API/library timed out or was aborted. - `transient_api`: temporary upstream/API failure suitable for retry/backoff. - `permanent_api`: unsupported, forbidden, not found, unauthorized, or bad-request condition that should not be retried blindly. - `parse_error`: source target/config could not be normalized into a public channel request. ## Compliance Notes - Allowed: public channel metadata and public message text collected through official APIs/libraries, with source review, clear legal notes, and PII minimization enabled by default. - Disallowed: private channel scraping, joining groups, account automation, invite-link/joinchat/private `t.me/c/` use, bypassing access controls, credential capture, CAPTCHA solving, and threat actor interaction. - PII minimization removes obvious email addresses and phone numbers from `rawText` before producing `CollectedItem`. More advanced redaction can be layered later in the evidence pipeline. - Public forwarded-message, reply/thread, pinned-message, and media fields are normalized as metadata. Media is metadata-only; the adapter does not download or retain raw Telegram files. - Extraction handoff metadata includes message text, quoted text, URL context, actor aliases, CVEs, victim candidates, and uncertainty markers for Agent 07. ## Operator Runbook - Onboard channels only from public URLs after source review records legal notes, approval scope, topic focus, language, and retention class. - Store API credentials outside `SourceRecord`; source metadata must never contain session strings, phone numbers, passwords, invite links, or user-account automation flags. - Runtime collection should use only active/approved `telegram_public` sources with `official_api` access. Do not run the adapter for candidate, pending-review, disabled, private, invite, or account-automation sources. - Respect `metadata.afterMessageId`, `metadata.beforeMessageId`, `metadata.rateLimitResetAt`, `metadata.channelCooldownUntil`, `metadata.publicQueryWindowLimit`, language hints, topic tags, and duplicate URL history before scheduling another channel window. - If runtime status is `rate_limited`, preserve `sourcePatch.crawlState.backoffUntil` and use `schedulerHints.retryAfterSeconds`; do not poll early. - If runtime status is `high_duplicate` or `high_churn`, reduce the next window, keep existing evidence immutable, and review whether the channel should be cooled down or suppressed for the query. - If runtime status is `policy_disabled`, `blocked`, or `unavailable`, stop new polling for that source and use apply-plan quarantine/review steps. Preserve only safe metadata, hashes, and provenance needed for audit. - Unsafe-source incident response: immediately disable the source, remove any credential/session/invite metadata, verify no raw media payloads were persisted, retain content hashes/provenance for audit, and request legal/source-owner review before restore. - On flood waits, update source crawl state or `metadata.rateLimitResetAt` and let the scheduler delay the next query window. - If a channel becomes private, invite-only, or starts requiring interaction, mark it disabled or quarantined and preserve only prior public evidence metadata. - For media-heavy posts, keep filename/mime/size/thumbnail hash metadata only; send no file-download task from this adapter.
