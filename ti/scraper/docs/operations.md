# Scraper Operations

## Runtime
- Bun + TypeScript.
- Start locally with `bun run start`.
- Test with `bun test`.
- Type-check with `bun run check` after dependencies are installed.
- Health: `GET /v1/health`.
- Metrics: `GET /v1/metrics`.
- Resource snapshot: `GET /v1/ops/resource-snapshot`.
- Run creation should use `idempotency-key` so retries do not duplicate work.

## Environment Config
- `SCRAPER_PORT`: API port, default `8097`.
- `SCRAPER_ENV`: `local`, `development`, `staging`, or `production`.
- `SCRAPER_MAX_REQUEST_TASKS`: per-request fanout cap, default `500`.
- `SCRAPER_MAX_TASK_BYTES`: per-task fetch cap, default `10485760`.
- `SCRAPER_CLEAR_WEB_CONCURRENCY`: global clear-web worker cap, default `64`.
- `SCRAPER_TELEGRAM_CONCURRENCY`: public Telegram worker cap, default `8`.
- `SCRAPER_DARKNET_METADATA_CONCURRENCY`: metadata-only darknet worker cap, default `4`.
- `SCRAPER_MEMORY_TARGET_MB`: default memory target, `98304` MB. Keep this under or equal to 96 GB unless explicitly approved.
- `SCRAPER_MEMORY_CEILING_MB`: normal ceiling, `163840` MB. Do not exceed 160 GB without reallocation.
- `SCRAPER_USER_AGENT`: clear-web collector user agent.
- `SCRAPER_DEFAULT_TIMEOUT_MS`: adapter timeout, default `30000`.
- `SCRAPER_HIGH_RISK_REQUIRES_APPROVAL`: default `true`.
- `SCRAPER_DARKNET_METADATA_ONLY`: default `true`.

## Inspur RAM Budget
- Start with 32 GB max for MVP collectors.
- Raise to 96 GB when multiple adapters and processing workers are stable.
- Reserve 160 GB as the normal scraper ceiling unless the TI leader reallocates.
- Dynamic browser crawling can consume memory quickly; keep Playwright disabled by default and isolate it in a separate worker pool.
- Disk-backed queues and object storage should carry bulk data. RAM caches must be bounded.

## Concurrency Draft
- RSS/static MVP: 32-128 concurrent fetches with per-source limits.
- Telegram: API-rate-limited, low concurrency.
- Tor/I2P/Freenet metadata: low concurrency, high timeout, strict allowlist, metadata-only.
- Processing: separate worker pool from collection.
- Dynamic browser pool: disabled by default. Enable only in a separate process/container with a hard worker cap and memory alarm.
- Queue: use bounded in-memory frontier for tests; production queue must be disk-backed before sustained crawling.

## Inspur Deployment Runbook
1. Build from a clean checkout and run `bun install`, `bun test`, and `bun run check`.
2. Start with `SCRAPER_ENV=production`, `SCRAPER_MEMORY_TARGET_MB=32768`, `SCRAPER_MEMORY_CEILING_MB=98304`, and conservative worker caps.
3. Put the scraper behind the CTI API/gateway. The scraper should trust only forwarded tenant/operator context from that layer.
4. Mount object storage and database volumes separately from application logs. Keep at least 500 GB free for the rest of the CTI application.
5. Enable clear-web/RSS workers first. Add Telegram and darknet metadata workers only after source approval fixtures pass.
6. Raise the memory target toward 96 GB only after queue age, worker failures, and duplicate rate stay stable for a full operating window.
7. Treat 160 GB as the normal ceiling, not a target. Reallocation above that needs leader approval.
8. Run `bun run check:docker-contexts` before deployment syncs. The root deploy context is capped at 40 MB; frontend, API, and scraper contexts are capped at 25 MB each.
9. Run `bun run check:deploy-hygiene` before building the live stack. This verifies required Docker files, scraper build test enforcement, healthcheck wiring, API dependency order, internal scraper URL, memory defaults, and secret/env-file context exclusions.

## Capacity Model
- MVP clear-web: 32 GB target, 64 clear-web workers, no browser workers, 0-2 darknet metadata workers.
- Stable mixed collection: 96 GB target, up to 128 clear-web workers, 8 Telegram workers, 4 darknet metadata workers.
- Normal ceiling: 160 GB total scraper ceiling including API, scheduler, adapters, processing workers, and caches.
- Emergency brake: if RSS exceeds 96 GB during live-search soak, stop browser workers first, set public search to queued-only, pause source-pack activation, and cut collection worker counts by half. If RSS approaches 160 GB, roll back scraper-native search and require explicit reallocation before continuing.
- 1 TB Inspur host reservation: keep scraper target at 96 GB and normal ceiling at 160 GB; reserve at least 500 GB RAM for API/frontend/search/DB/graph/model workloads and leave the remaining RAM as OS/page-cache/object-store headroom. Browser workers stay zero unless explicitly reallocated.
- Full CTI RAM reservation on a 1 TB host: scraper 96 GB target and 160 GB normal ceiling; API 16-32 GB; frontend 4-8 GB; Postgres 64-128 GB; OpenSearch/vector 128-192 GB; graph store 64-128 GB; object store/page cache 128-192 GB; OS cache plus emergency spare 160-256 GB. Keep at least 500 GB outside the scraper budget for the broader CTI app and model/search workloads before raising scraper workers.
- Browser crawling: budget separately at the process/container level. Do not borrow from the clear-web pool without reducing other workers.
- RAM caches are for hot metadata only. Raw evidence, captures, and queues should move to disk/object storage before scale testing.
- Worker-pool sizing must account for clear-web, processing, Telegram, browser, and darknet metadata pools separately. Keep browser workers at zero by default.
- Darknet metadata workers must remain low-concurrency; counts above 8 are treated as a configuration error.
- Capacity estimates include API baseline, per-worker memory, queue memory, and a safety margin. If the estimate exceeds the 96 GB target, reduce browser, clear-web, or processing workers before considering more RAM.

## Backup And Restore
- Postgres: nightly logical backup for sources, tasks, runs, captures metadata, incidents, audit events, and dedupe indexes.
- Object storage: immutable raw evidence bucket with versioning where allowed by policy. Metadata-only captures must never rely on raw leaked payload backups.
- Restore drill: monthly restore into an isolated environment, then run capture replay through the current pipeline version and compare incident counts.
- Keep backup credentials outside source records and outside adapter config.

## Retention Jobs
- `restricted_metadata`: keep metadata and hashes; purge any accidental body fields immediately.
- `short`: purge raw body/object references on the configured short window, retaining provenance and hashes.
- `standard`: retain according to tenant/source policy and storage budget.
- `legal_hold`: exclude from automated purge until governance clears the hold.
- Retention jobs must log counts by tenant, source, retention class, and action.

## Darknet Metadata Runbook
- Onboard Tor/I2P/Freenet sources only as `*_metadata` source types with `risk: high`, `accessMethod: approved_proxy`, legal/ethics notes, and `approvedAt`/`approvedBy` recorded.
- New darknet metadata seeds should start `disabled`/`needs_review` with pending metadata-only governance. Activate only after legal/ethics review, operator approval, parser fixture review, and proxy boundary assignment.
- Route collection through an approved proxy boundary object for the matching network. Use network-specific metadata config for max metadata bytes, request timeout, concurrency, allowed schemes, and screenshot hash mode. Do not put proxy routing, credentials, or evasion settings in source records.
- Capture only actor, victim, claim date, sector, country, claimed data type, source timestamp, URL hash, policy decision, and screenshot hash when screenshot hash mode is `hash_only`. Store leak-site items as sensitive metadata-only captures and never persist screenshot bytes by default.
- Block download-like URLs, credential-bearing URLs, archive/database/document/media extensions, forms, login/auth prompts, payment/contact flows, comments, uploads, interactive search, CAPTCHA solving, credential bypass, threat actor interaction, stealth behavior, and any request for stolen payload bodies.
- Keep concurrency low and source-scoped. Treat repeated policy blocks as an alertable operator event, not as retryable crawl failures.
- Review new parsers against saved safe fixtures before enabling them for live metadata sources.
- Agent 02 coordination: darknet metadata sweeps need restricted request budgets, source-scoped concurrency keys, short retry budgets, and no destination expansion into payload/download paths.
- Agent 09 API coordination: live search darknet metadata states are `disabled`, `approval_required`, `queued_metadata_only`, `partial_metadata`, and `blocked`. Result DTOs expose parsed metadata, hashes, confidence, and policy audit id only.
- Agent 06 retention handoff: metadata-only evidence should persist hashes, source timestamps, post status, confidence, policy decisions, and parsed claim fields under `restricted_metadata`; default TTL should be short unless legal hold is set. Any accidental body/object reference must be rejected or purged immediately.
- Agent 10 isolation handoff: run Tor/I2P/Freenet metadata workers in separate low-concurrency worker pools with independent egress controls, timeout classes, memory caps, proxy/circuit attribution, and policy-block metrics.
- Kill switch: setting a source status to `disabled` or access method to `disabled` must block collection immediately; a global worker-pool kill switch should stop all darknet metadata workers before any retry/requeue and make `/v1/intel/search` return darknet metadata status `disabled` with an explainable blocked entry.
- Evidence deltas for darknet metadata are polling explanations, not proof of compromise. Supported restricted delta reasons are `newly_observed_claim`, `changed_claim_status`, `mirrored_post`, `removed_dead_post`, `blocked_unsafe_link`, and `policy_change`; each delta must stay `restricted_metadata` and include parsed claim fields, hashes, policy audit ids, and source ids only.
- Analyst interpretation labels are intentionally cautious: `unverified_actor_claim` means a source asserted the claim, `mirrored_claim` means another source repeated it, `stale_claim` means the source timestamp is old, `contradicted_claim` means another signal conflicts with it, and `needs_analyst_review` means it should not drive automated conclusions. None of these labels confirm intrusion, exfiltration, victim identity, or actor attribution.
- Do not alert or graph-promote raw leak material from restricted metadata. Agent 08 graph consumers should downgrade or contest relationships when metadata-only claims are mirrored, stale, contradicted, or unreviewed. Agent 10 alerting should report policy blocks, kill switches, and approval-change deltas using counts, source ids, hashes, and policy reasons only.
- Restricted approval bridge states are API-visible for `/v1/intel/search`: `disabled_kill_switch`, `missing_proxy_approval`, `missing_legal_notes`, `pending_metadata_only_approval`, `active_metadata_only_queue`, and `blocked_unsafe_target`. These map to public live-search states without creating collection tasks unless the source is active, approved, metadata-only, legally noted, assigned to a matching approved proxy boundary, and the target URL passes the unsafe-link blockers.
- A reviewer approves only the source identity, legal/ethics basis, matching proxy boundary, metadata-only governance, risk justification, review ticket/expiry, and the narrow parser output fields: actor, victim, claim date, sector, country, claimed data category, post status, source timestamp, URL hash, screenshot hash, confidence, and policy audit id.
- Approval never permits stolen-file download, credential bypass, CAPTCHA solving, stealth/evasion, private access, threat actor interaction, comments/forms/uploads/contact/payment flows, interactive search, raw leaked bodies, screenshot bytes, media/archive/database/document retrieval, or broad destination expansion. Those remain impossible operations and should stay visible in bridge DTOs and runbook reviews.
- Agent 01 lifecycle coordination: `candidate` and `needs_review` remain `pending_metadata_only_approval`; an `approve` decision must explicitly set `metadataOnly: true`; `approved` still does not queue until an operator transitions to `active`; `disabled` is a kill-switch state; `restore` may return to `active` only if approval remains metadata-only and source health is acceptable.
- Agent 09 API coordination: preserve bridge `sourceStates`, blocked `restrictedState`, `requiredAction`, `metadataOnly`, and reason fields in search responses so the UI can explain whether the next action is legal notes, proxy assignment, metadata-only approval, restore, or target repair. Agent 10 should alert on repeated `blocked_unsafe_target`, kill-switch transitions, and missing proxy approval counts without raw URLs.
- Restricted metadata source packs are dry-run review artifacts. A pack must be `disabledByDefault`, network-scoped to Tor/I2P/Freenet metadata, bound to a named approved proxy boundary, assigned to a kill-switch group, carry legal notes, use approval scope `metadata_only`, declare `restricted_metadata`/`darknet_metadata`/`legal_hold` retention, and list every forbidden operation.
- Pack validation must not make network requests or activate sources. It produces reviewer reports showing the exact metadata fields that would be captured, operations that will never be captured, approval prerequisites, blocked unsafe targets, alert names that would fire, and API notes for Agent 09 so recommendations do not look like raw leak collection.
- Unsafe pack examples should remain fixtures only: download-heavy leak sites, credential dump paths, auth-only forums, contact/payment pages, media-heavy sources, and a metadata-only listing. Unsafe examples should fire policy-block or kill-switch-group report entries while imported source records stay `needs_review` with `accessMethod: disabled`.
- Agent 06 retention coordination: default restricted pack outputs to `restricted_metadata`; use `legal_hold` only when a reviewer records a legal hold reason. Agent 10 kill-switch coordination: every pack and entry must include a low-cardinality kill-switch group suitable for alert labels and emergency disablement.
- Restricted operations bridge outputs must stay API-ready and metadata-only. Allowed fields are exactly actor/group name, victim name, claimed date, claimed sector, claimed country, claimed data type/category, URL hash, screenshot hash, source timestamp, post/status, confidence, policy audit id, and provenance ids. Status contracts may expose `disabled_kill_switch`, `missing_proxy_approval`, `missing_legal_notes`, `pending_metadata_only_approval`, `active_metadata_only_queue`, `blocked_unsafe_target`, and `retention_expiry`.
- Restricted activation recommendations for actor, victim, and ransomware queries must never include raw source URLs, credentials, private forum content, stolen files, or payload paths. Show source id/name, required action, matched safe metadata fields, allowed field names, forbidden operations, and a reason only. Retention expiry should emit an `expired` evidence delta with `deltaType: retention_expiry` and no raw content.

## Live Search SLOs
- Initial `/ti` response target: 1 second.
- First partial-result target: 5 seconds.
- Normal polling interval: 2 seconds; degraded polling interval: 10 seconds.
- Active-run limit: one active run per tenant/query. Reuse existing runs instead of creating duplicate live-search work.
- Provider failure budget: 2% of live-search requests.
- Zero-result budget: 25% of live-search requests before source activation gaps need review.
- Public `/ti` backpressure should reject or delay new runs when a tenant/query is already searching, the scraper/API proxy is degraded, the queue is backed up, no approved sources are available, or a darknet metadata kill switch is active.
- Live-search metrics should include provider failures, zero-result searches, duplicate active runs, polling volume, source activation gaps, and external dependency latency. Metric labels must stay low-cardinality and avoid raw query strings.
- Additional public-path metrics: fallback provider health, scraper-native `/v1/intel/search` health, temporary outer-wrapper fallback usage, and source-activation gaps.
- Polling impact at 2 seconds: 1 client is 0.5 req/s and about 65 MB ops overhead; 10 clients are 5 req/s and about 69 MB; 100 clients are 50 req/s and about 119 MB; 1,000 clients are 500 req/s and about 505 MB. CPU/API request pressure, not RAM, is the limiting factor at high polling counts.
- Slow polling when duplicate active runs appear, queue backlog grows, or provider latency exceeds the partial-result SLO. Disable external live discovery when the fallback provider is down, starts dominating responses, or consumes the failure budget. Return queued-only when the scraper is healthy but providers are degraded. Require login/API key when anonymous clients create sustained high polling volume, source activation probing, or repeated zero-result fanout.
- The outer fallback may be removed only when the scraper-native readiness gate passes: scraper endpoint healthy, random actor query returns partial evidence, run id present, cursor polling works, source coverage returned, degraded responses remain controlled, and public `/ti` proof still shows live partials.
- Roll back to the outer fallback if scraper-native `/v1/intel/search` returns non-2xx, omits run ids, breaks cursor polling, loses source coverage explanations, reports uncontrolled degraded/error states, or causes public `/ti` to stop showing live partial markers.

## Incident Runbooks
- High memory: stop browser/dynamic workers first, lower collection concurrency, inspect queue size, then raise memory only if leader-approved.
- Queue backlog: check source health, adapter errors, downstream storage latency, and per-source concurrency locks before adding workers.
- Source breakage: move the source to `degraded` or `paused`, preserve last successful capture metadata, and open parser fixture work.
- Unsafe collection attempt: leave the source disabled or review-required, preserve the policy decision/audit event, and do not retry automatically.

## Main-Agent Alert Response
- Memory alert: fetch `/v1/ops/resource-snapshot`, compare current RSS to target, inspect worker pool sizing, set browser workers to zero if any are active, then reduce clear-web and processing concurrency until capacity is back under target. Do not raise above 160 GB without explicit reallocation.
- Queue age alert: inspect queued and leased counts, source health, retry/backoff state, and downstream storage latency. Add workers only if the capacity estimate remains below the 96 GB target and source-level limits are not the bottleneck.
- Adapter failure-rate alert: identify adapter and source IDs from structured logs, pause or degrade repeatedly failing sources, preserve warnings/last-success metadata, and create fixture coverage before re-enabling.
- Telegram rate-limit alert: inspect `scraper_adapter_rate_limit_delay_seconds`, delay the source window until `rateLimitResetAt`, and avoid increasing Telegram concurrency while the alert is active.
- Live search degraded: check `scraper_live_search_provider_failures`, `scraper_live_search_external_dependency_latency_ms`, and `/v1/ops/resource-snapshot`; then switch clients to the degraded poll interval and cached partials until provider failures clear.
- Live search wrapper fallback: check `scraper_live_search_outer_fallback_used`, fallback-provider health, and scraper-native endpoint health. Prefer scraper-native `/v1/intel/search`; keep the outer fallback temporary and alert while it is serving production traffic.
- Scraper-native cutover failure: restore the previous `api/src/utils/ti/search.ts` fallback path, redeploy `hanasand_api`, run `bun run check:live-search-deploy`, then keep `scraper_live_search_scraper_native_healthy=0` alerting until `bun run check:scraper-native-search` passes again.
- Cursor polling broken: stop new scraper-native cutover attempts, return queued-only or cached partial responses, and verify cursor fields plus delta payloads with Agent 09 before reenabling public polling.
- Source activation gap too high: keep public search in partial/blocked mode, surface activation recommendations, and ask Agent 01/02 to review source coverage before increasing fanout.
- 24h soak rollback: set the API wrapper back to the last known-good fallback, redeploy `hanasand_api`, verify `curl -fsS https://api.hanasand.com/api/ti/search?q=APT29`, run `bun run check:live-search-deploy`, then post the soak status template with rollback reasons to the leader thread and `coordination.md`.
- 24h soak escalation: page the main agent when RSS exceeds 96 GB, queue p95 exceeds 120s, worker saturation exceeds 95%, scraper-native health fails twice in a row, public proof fails, unsafe policy retries appear, stale-cache rate exceeds 10%, source-unavailable rate exceeds 20%, or duplicate active runs reappear after Agent 02 backpressure.
- Public-channel polling kill switch: disable public-channel source-pack activation, keep existing approved captures read-only, raise poll interval to 10 seconds, and require login/API key if anonymous public polling keeps pressure high.
- Restricted metadata kill switch: set restricted sources to `disabled` or keep `SCRAPER_DARKNET_METADATA_ONLY=true` with no active metadata workers; never retry policy-blocked restricted targets.
- Source-pack activation rollback: revert newly activated source packs to `candidate` or `needs_review`, keep provenance and activation dry-run logs, and rerun source activation coverage before restoring.
- Live search no approved sources: surface source activation recommendations, keep the run in `blocked` or `partial`, and do not bypass governance for Telegram, darknet, forums, or leak-site material.
- Live search queue backlog: reuse active runs, stop creating duplicate tenant/query runs, raise polling interval to 10 seconds, and reduce fanout before adding workers.
- Live search darknet kill switch: keep darknet metadata tasks disabled, return clear public-source partials only, and record the blocked state as an operational event rather than a retryable failure.
- Policy-block alert: verify the policy decision, keep high-risk or restricted sources review-gated, and do not convert blocks into automatic retries.
- Docker context drift alert: inspect `.dockerignore` first, then remove generated build artifacts from the context. Do not ship contexts containing `node_modules`, `.next`, coverage, or large unrelated directories.
