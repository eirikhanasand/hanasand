# Scraper Operations

## Runtime
- Bun + TypeScript.
- Start locally with `bun run start`.
- Test with `bun test`.
- Type-check with `bun run check` after dependencies are installed.
- Health: `GET /v1/health`.
- Metrics: `GET /v1/metrics`.
- Resource snapshot: `GET /v1/ops/resource-snapshot`.
- Run creation should use `idempotency-key` so retries do not duplicate work.

## Environment Config
- `SCRAPER_PORT`: API port, default `8097`.
- `SCRAPER_ENV`: `local`, `development`, `staging`, or `production`.
- `SCRAPER_MAX_REQUEST_TASKS`: per-request fanout cap, default `500`.
- `SCRAPER_MAX_TASK_BYTES`: per-task fetch cap, default `10485760`.
- `SCRAPER_CLEAR_WEB_CONCURRENCY`: global clear-web worker cap, default `64`.
- `SCRAPER_TELEGRAM_CONCURRENCY`: public Telegram worker cap, default `8`.
- `SCRAPER_DARKNET_METADATA_CONCURRENCY`: metadata-only darknet worker cap, default `4`.
- `SCRAPER_MEMORY_TARGET_MB`: default memory target, `98304` MB. Keep this under or equal to 96 GB unless explicitly approved.
- `SCRAPER_MEMORY_CEILING_MB`: normal ceiling, `163840` MB. Do not exceed 160 GB without reallocation.
- `SCRAPER_USER_AGENT`: clear-web collector user agent.
- `SCRAPER_DEFAULT_TIMEOUT_MS`: adapter timeout, default `30000`.
- `SCRAPER_HIGH_RISK_REQUIRES_APPROVAL`: default `true`.
- `SCRAPER_DARKNET_METADATA_ONLY`: default `true`.
- `TI_CANARY_ENABLED`: starts the bounded public canary loop unless set to `false`.
- `TI_CANARY_AUTO_ACTIVATE`: default `false`. Keep it false in production so the loop collects only sources already activated by `POST /v1/sources/canary-activation`.
- `TI_CANARY_INTERVAL_SECONDS`, `TI_CANARY_MAX_TASKS`, `TI_CANARY_MAX_SOURCES`, `TI_CANARY_MAX_QUEUE_SIZE`, `TI_CANARY_BUDGET_TASKS`, and `TI_CANARY_BUDGET_BYTES`: bound public canary cadence, fanout, queue pressure, and byte budgets.
- `TI_EVIDENCE_ROOT`: production evidence root, default `/var/lib/ti-scraper/evidence` when `SCRAPER_ENV=production`.
- `TI_EVIDENCE_METADATA_PATH` and `TI_EVIDENCE_OBJECT_DIR`: override the file-backed metadata snapshot and object evidence directory. In production these should stay under the mounted evidence volume, not `/tmp`.

## Inspur RAM Budget
- Start with 32 GB max for MVP collectors.
- Raise to 96 GB when multiple adapters and processing workers are stable.
- Reserve 160 GB as the normal scraper ceiling unless the TI leader reallocates.
- Dynamic browser crawling can consume memory quickly; keep Playwright disabled by default and isolate it in a separate worker pool.
- Disk-backed queues and object storage should carry bulk data. RAM caches must be bounded.

## Concurrency Draft
- RSS/static MVP: 32-128 concurrent fetches with per-source limits.
- Telegram: API-rate-limited, low concurrency.
- Tor/I2P/Freenet metadata: low concurrency, high timeout, strict allowlist, metadata-only.
- Processing: separate worker pool from collection.
- Dynamic browser pool: disabled by default. Enable only in a separate process/container with a hard worker cap and memory alarm.
- Queue: use bounded in-memory frontier for tests; production queue must be disk-backed before sustained crawling.

## Public Adapter Capture Contracts
- Dynamic public web capture uses a Playwright-compatible renderer boundary but remains disabled by default in production worker planning. Tests use injected renderers only; live browser work must stay public, unauthenticated, non-CAPTCHA, and isolated from restricted adapters.
- PDF/report capture uses an injected extractor boundary for public vendor reports, advisories, and long-form writeups. Collected items preserve canonical URL, publication time, content hash, source trust, extraction status, parser warnings, citation spans, and provenance.
- Parser profiles are selected from `static_html`, `dynamic_page`, `pdf_report`, `rss_entry`, and `public_channel_handoff` by source type, media type, JavaScript requirement, and handoff metadata. The matrix emits fallback order, fallback reasons, parser score, confidence band, language hint, and expected media types.
- Agent handoff DTOs must be redaction-safe. Agent 06 receives capture readiness, canonical URL, content hash, and provenance requirements; Agent 07 receives parser profile, score, confidence band, language hint, citation spans, and parser warnings without raw text/html fields; Agent 09 receives API status and retryability; Agent 10 receives adapter cost class and dashboard state.
- Dynamic/PDF failure taxonomy is limited to safe categories: `timeout`, `robots_policy_hold`, `policy_hold`, `unsupported_media`, `content_too_large`, `parser_confidence_low`, `duplicate_canonical`, `rate_limited`, `unavailable`, and `source_disabled`.
- Cost expectations: RSS/static are low cost, PDF/report extraction is medium cost, and dynamic page rendering is high cost. Dynamic capture should be queued behind static/PDF/RSS work unless a source profile proves static extraction is insufficient.
- Source freshness regression should run before promoting new public sources. It classifies stale feeds, broken parser profiles, empty captures, noisy sources, duplicate-heavy sources, and policy-disabled sources into keep-active, review, or disable recommendations.
- Browser worker readiness is a contract layer until explicitly enabled: the dynamic browser pool reports public-only network isolation, private-network/credential/CAPTCHA/download blocking, memory and timeout caps, robots/legal policy state, and hash-only screenshot evidence.
- Report corpus benchmarks use `ti.report_corpus_benchmark.v1` for Program BB public report/PDF/OCR extraction readiness. Fixtures should cover vendor reports, CERT/vendor advisories, multilingual reports, RSS items, dynamic snapshots, scanned PDFs, malformed PDFs, unsupported MIME, stale reports, duplicate canonical reports, and policy-held restricted targets. Benchmark rows score parser quality, language readiness, citation-span coverage, media support, stale publication dates, OCR readiness, `CollectedItem`-compatible provenance, and extraction readiness without raw payload fields.
- Report extraction readiness rows emit pass/watch/hold handoffs for Agent 03 parser/certification work, Agent 06 capture replay, Agent 07 quality gates, Agent 09 route output, and Agent 10 release gates. They preserve source id, task id, canonical URL hash, content hash, fetched-at, published-at, robots/legal notes, language, parser confidence, citation spans, and duplicate-canonical keys while forbidding raw URLs, unsafe URLs, raw text, HTML, bodies, payloads, object refs/keys, OCR vendor IDs, credentials, cookies, tokens, private invites, and onion URLs.
- OCR readiness rows classify text-layer presence, scanned-page count, OCR availability/confidence, citation-span coverage for scanned reports, and pass/watch/hold handoffs. `not_required` and `ready` rows can count toward production report coverage; `needs_ocr` rows should request an Agent 03 OCR/parser fixture before broad rollout; `blocked` rows should hold PDF promotion and evidence replay until text-layer or OCR confidence is sufficient.
- Adapter failure observatory packets use `ti.adapter_failure_observation.v1`, `ti.adapter_failure_observatory.v1`, and `ti.adapter_production_readiness_packet.v1`. They cover static HTML, RSS, dynamic-page contracts, public PDF/report capture, public-channel handoffs, and advisory/security signals. Route-visible fields include source id/type/family, query class, parser profile/version, safe failure class, outcome, canonical URL hash, retry-after, stale date, robots/legal hold, unsupported MIME, content-too-large, timeout, duplicate canonical, rate-limit/unavailable/source-disabled flags, parser confidence, extraction warnings, and explicit Agent 01/02/06/07/09/10 handoff actions.
- Observatory DTOs are API-ready and no-leak by design: raw URLs, unsafe onion links, HTML, raw text, payloads, credentials, cookies, tokens, private invites, and restricted raw material are forbidden. Dynamic browser readiness remains a disabled planning packet until separately allocated; public-only, no auth bypass, no CAPTCHA solving, no private communities, no exploit payload download, and no restricted raw material remain hard defaults.
- Adapter runtime enablement packets use `ti.adapter_runtime_enablement_packet.v1` and per-adapter `ti.adapter_runtime_readiness.v1` rows. They cover static HTML, RSS, dynamic public browser, PDF/report, public-channel handoff, and advisory/security adapters with canary source ids, canonical URL hashes, pool caps, memory caps, timeout caps, byte caps, parser confidence thresholds, evidence-yield gates, rollback triggers, and Agent 01/02/06/07/09/10 handoffs. Static/RSS/advisory can report default-enabled only when gates pass; dynamic browser, PDF/report, and public-channel handoffs stay canary-only until a separate operator allocation promotes them.
- Production adapter runtime program packets use `ti.production_adapter_runtime_program.v1` and per-adapter `ti.production_adapter_capability.v1` rows to merge runtime enablement, parser certification, SLA repair, and repair triage into one operator view. The packet covers RSS, static HTML, public PDF/report extraction, disabled-by-default dynamic public browser isolation, public-channel handoffs, advisory signals, GitHub/security advisory feeds, and multilingual handoffs. Each capability row reports runtime mode, implementation state, parser certification state, source family, parser profile, byte/time/worker limits, language support, robots/legal requirements, canonical/content hash capture requirements, evidence replay refs, retry/backoff support, conditional-request support, fixture replay gaps, failure classes, blockers, warnings, and Agent 01/02/04/06/07/09/10 handoffs.
- Production capture metadata must be extraction-ready but route-safe: `sourceId`, `canonicalUrlHash`, `contentHash`, `fetchedAt`, `language`, `parserConfidence`, `extractionWarnings`, provenance, and `evidenceReplayRef` are required. Raw URLs, requested/final URLs, raw text, translated text, HTML, bodies, payloads, credentials, cookies, tokens, private invites, onion links, screenshot bytes, download URLs, object refs, and object keys remain forbidden in public API output.
- Public advisory runtime capture uses `PublicAdvisoryAdapter` for approved public API/HTTP advisory feeds, including GitHub Security Advisory-style JSON, CISA/KEV-style JSON, CERT/government feeds, vendor advisory feeds, malware/tool feeds, public report indexes, and RSS/Atom advisory entries. It normalizes records into `CollectedItem` with source id, canonical URL, content hash, published/observed timestamps, advisory source family, parser confidence, extraction warnings, matched CVE/actor/malware/tool/campaign/sector/country/victim tags, deterministic dedupe keys, and `evidenceReplayRef`.
- Advisory route/API handoff should use the embedded `ti.public_advisory_signal_delta.v1` object rather than raw item fields. The safe delta exposes source id, source family, title, canonical URL hash, content hash, timestamps, confidence, matched entities, dedupe key, replay ref, and public-only provenance. It must not expose raw URLs, raw text, HTML, bodies, payloads, credentials, cookies, tokens, private invites, onion links, download URLs, object refs, or object keys.
- Advisory failure handling is fail-closed for private repository advisory paths, auth/private/login/invite paths, secret-bearing URLs, payload/download/exploit/PoC affordances, malformed schemes, non-HTTP schemes, and onion links. Suppressed records remain visible only as title, family, reason, and unsafe URL hash so operators can audit why they did not contribute. `304` responses produce `not_modified`; `429` produces `rate_limited`; missing publication dates and thin records become extraction warnings for Agent 07/10 gates.
- Live capture runtime packets use `ti.live_capture_runtime_packet.v1` and `ti.live_capture_runtime_row.v1` to harden repeated RSS/static/report-index/public-advisory runs before evidence replay. Rows include adapter, capture status, safe failure class, canonical URL hash, content hash, dedupe key, replay ID, parser confidence, extraction warnings, freshness state, timeout/content-size/MIME/redirect caps, HTTP/retry/robots/legal diagnostics, Agent 01 source-pack readiness, Agent 02 cadence hints, and Agent 06 evidence handoff descriptors. They are dry-run route contracts over already-approved adapter results and do not start unapproved live crawling.
- Agent 06 handoff descriptors from live capture runtime use `ti.live_capture_evidence_handoff.v1` with raw capture descriptor, text projection descriptor, source metadata, extraction version, claim candidate IDs, and replay ID. Descriptors are hash/provenance only: raw URLs, raw text, HTML, bodies, payloads, credentials, private invites, onion links, download URLs, object refs, and object keys stay forbidden in route/API output.
- Live capture conformance fixtures should cover `github_security_advisory`, `cisa_kev`, `vendor_advisory_json`, `cert_html`, `vendor_blog_html`, `rss_atom`, and `report_index`. Missing fixture classes are release/watch signals for Agent 03; duplicate content lowers cadence through Agent 02; policy/unsafe URL holds pause scheduling and hold source activation through Agent 01.
- Live capture canary packets use `ti.live_capture_canary_packet.v1` and `ti.live_capture_canary_row.v1` for disabled-by-default fixture replay or dry-run canaries across RSS, static HTML, report indexes, public advisories, and PDF text-layer reports. Rows expose source id, adapter family, approved URL hash, robots/legal note state, timeout/content/MIME/redirect caps, parser version, extraction warnings, dedupe hashes, evidence replay refs, no-leak policy result, canary state, parser repair recommendation, and scheduler hint.
- Parser repair recommendations classify malformed feeds, changed static layouts, report-index drift, public advisory schema changes, unsupported MIME, excessive redirects, source outage, duplicate-heavy output, stale source windows, and hostile/unsafe-link suppression. Canary promotion states are `promote`, `watch`, `hold`, and `rollback`; scheduler hints cover first-run, repeat-run, burst-failure, source-outage, parser-regression, and source-family-shortage cases without mutating source status or leasing queue work.
- Canary conformance should add PDF text-layer reports, unsupported MIME, and hostile/unsafe-link suppression to the live capture fixture set before promotion. The packet is route-safe and proof-only: it must report `willStartNetworkCollection:false`, `willMutateSources:false`, `willLeaseQueueWork:false`, no raw URLs/text/HTML/PDF bytes/object keys, and no unsafe URL exposure.
- PDF/report extraction readiness uses `ti.pdf_report_extraction_readiness.v1` before public PDF report captures are promoted into evidence replay. Rows keep OCR disabled by default, expose only text projection hashes, citation-span counts, parser confidence, extraction warnings, language, canonical URL hash, content hash, and `evidenceReplayRef`, and report pass/watch/hold status for Agent 03 parser work, Agent 06 replay, Agent 07 quality gates, Agent 09 route output, and Agent 10 release gates.
- PDF/report readiness is text-only and route-safe. OCR may be requested only through a separate operator approval path; readiness packets must not expose raw PDF bytes, raw text, object references, object keys, OCR vendor IDs, credentials, cookies, tokens, unsafe URLs, onion links, or restricted material. Low confidence, missing citation spans, parse failures, unsupported media, legal/robots holds, or unavailable sources should keep the row in watch/hold instead of silently promoting report evidence.
- Dynamic browser rollout remains disabled by default even in canary-ready packets. Browser evidence stores screenshot hashes only, never screenshot bytes; failures or attempted raw screenshot/URL serialization are rollback triggers. Operators should hold or roll back on policy holds, high blocked/watch failure ratios, parser confidence below threshold, byte-cap breaches, timeout spikes, memory-cap breaches, low evidence yield, or unsafe serialization.
- Dynamic browser isolation cutover packets use `ti.dynamic_browser_cutover.v1` for public pages that static/RSS adapters cannot capture. The packet is disabled-by-default, canary-only, and requires explicit approval; it reports worker-pool size, memory caps, render timeout caps, byte caps, queue caps, public host allowlist hashes, screenshot-hash-only evidence, object-ref hashes, fixture-replay-only isolation canaries, and kill-switch rollback state. Program BC adds `resourceBudget`, `storageIsolation`, per-fixture `provenance`, `browser_pool_isolated`, `storage_ephemeral`, and `promotionReadiness`: browser work must stay in a separate worker pool, cookies/local/session/cache/downloads/screenshot bytes/raw HTML/object refs are never persisted or exposed, each fixture preserves source id, task id, canonical/final/requested URL hashes, content hash, fetched-at, parser confidence, extraction warnings, robots/legal notes, screenshot hash, and object-ref hash, and live browser enablement remains `disabled_requires_separate_operator_allocation`. Operators must hold on JS render timeout, redirect-chain overflow, unsupported MIME, robots/legal hold, capture truncation, blank page, parser empty extraction, screenshot hash mismatch, queue pressure, private-network targets, credential prompts, CAPTCHA challenges, download attempts, onion redirects, third-party request leaks, missing approval, kill-switch activation, resource overrun, or storage isolation drift. Agent handoffs feed Agent 01 source activation, Agent 02 scheduler budgets, Agent 04 public expansion, Agent 06 evidence chain, Agent 07 quality gates, Agent 09 API warning fields, and Agent 10 resource gates without raw screenshots, raw HTML, raw URLs, onion links, private material, credentials, payloads, cookie jars, local storage, or object refs in API output.
- Multilingual public report handoffs use `ti.translation_handoff.v1` for public HTML, RSS, dynamic-page captures, PDF/report text, advisory/security signals, and public-channel handoffs. The packet is vendor-neutral: it records declared/detected/original/target/requested language, translation-needed state, mixed-language signals, citation spans in original offsets, parser fallback order, adjusted extraction confidence, source language scoring, and Agent 01/04/06/07/09/10 handoffs without adding a live translation service dependency.
- Multilingual parser confidence benchmarks use `ti.multilingual_parser_confidence_benchmark.v1` to roll translation handoff packets into pass/watch/hold rows by original language. Benchmarks track adjusted parser confidence, detection confidence, citation-span coverage, translation-needed counts, mixed-language ratio, high-priority translation ratio, low-confidence blocks, packet refs, thresholds, and Agent 01/04/06/07/09/10 handoffs without raw text, raw URLs, translated text, or vendor coupling.
- Multilingual handoff DTOs must never expose raw text, translated text, HTML, raw URLs, unsafe onion links, translation vendor names, API keys, credentials, cookies, payloads, private invites, or restricted raw material. Agent 06 should retain original-language metadata and translation metadata only; Agent 07 should review translation-needed and mixed-language reports before high-confidence extraction promotion.
- Adapter SLA enforcement and parser repair packets use `ti.adapter_sla_repair_packet.v1`, `ti.adapter_sla_contract.v1`, and `ti.adapter_parser_repair.v1`. They define production SLA rows for static HTML, RSS, dynamic public browser, PDF/report, public-channel handoff, advisory/security signal, and multilingual handoff adapters, tracking parser confidence, extraction warnings, translation needs, unsupported MIME, timeout/rate-limit/unavailable states, stale dates, canonical URL hashes, and duplicate canonical keys.
- Parser repair packets classify fixture gaps, selector failures, readability failures, PDF extraction failures, language detection drift, dynamic-render failures, unsupported MIME, scheduler backoff, and duplicate evidence suppression. Repairs are dry-run, hash-only, and API-safe: they hand source governance to Agent 01, backoff/cadence to Agent 02, confidence/correlation to Agent 04, replay/suppression to Agent 06, extraction quality to Agent 07, warning codes to Agent 09, and release holds/watch runbooks to Agent 10 without exposing raw URLs, raw text, HTML, screenshot bytes, credentials, private invites, onion links, or restricted material.
- Adapter repair triage packets use `ti.adapter_repair_triage_packet.v1` to rank whether each public adapter/source should be certified, repaired, paused/disabled, cadence-reduced, duplicate-suppressed, or escalated to a release hold. Triage consumes SLA repair packets plus certification replay gates and scores customer-visible search impact, source-family coverage, parser confidence, freshness debt, duplicate rate, unsupported MIME, timeout/rate-limit patterns, language drift, and certification gaps. Sandbox replay expectations stay hash-only and disabled-by-default for dynamic browser work; dynamic public rendering still requires explicit approval. Triage is advisory and dry-run only, with Agent 01/02/04/06/07/09/10 handoffs and no raw URLs, raw text, HTML, screenshots, object refs, credentials, private invites, onion links, or restricted material in route/API output.
- Adapter runtime certification uses `ti.adapter_certification_packet.v1` and per-fixture `ti.adapter_fixture_replay.v1` rows before public adapters can be trusted for live actor-query collection. Fixture replay must cover static HTML, RSS, dynamic public browser, PDF/report, public-channel handoff, advisory/security signal, and multilingual handoff adapters across success, parser drift, stale dates, language mismatch, unsupported MIME, timeout, rate-limit, duplicate canonical, truncated capture, and empty-extraction cases.
- Certification output is evidence-safe by contract: fixture URLs and object references are hashed, content is represented by hashes, and extraction stats are limited to byte counts, text length, link count, citation-span count, and warning count. Dynamic public browser fixtures remain uncertified unless `explicitBoundedDynamic` is true; even then the packet reports `browserWorkersEnabled: false` and is a dry-run certification artifact, not a worker enablement switch.

## Inspur Deployment Runbook
1. Build from a clean checkout and run `bun install`, `bun test`, and `bun run check`.
2. Start with `SCRAPER_ENV=production`, `SCRAPER_MEMORY_TARGET_MB=32768`, `SCRAPER_MEMORY_CEILING_MB=98304`, and conservative worker caps.
3. Put the scraper behind the CTI API/gateway. The scraper should trust only forwarded tenant/operator context from that layer.
4. Mount object storage and database volumes separately from application logs. Keep at least 500 GB free for the rest of the CTI application.
5. Enable clear-web/RSS workers first. Add Telegram and darknet metadata workers only after source approval fixtures pass.
6. Raise the memory target toward 96 GB only after queue age, worker failures, and duplicate rate stay stable for a full operating window.
7. Treat 160 GB as the normal ceiling, not a target. Reallocation above that needs leader approval.
8. Run `bun run check:docker-contexts` before deployment syncs. The root deploy context is capped at 40 MB; frontend, API, and scraper contexts are capped at 25 MB each.
9. Run `bun run check:deploy-hygiene` before building the live stack. This verifies required Docker files, scraper build test enforcement, healthcheck wiring, API dependency order, internal scraper URL, memory defaults, and secret/env-file context exclusions.

## Capacity Model
- MVP clear-web: 32 GB target, 64 clear-web workers, no browser workers, 0-2 darknet metadata workers.
- Stable mixed collection: 96 GB target, up to 128 clear-web workers, 8 Telegram workers, 4 darknet metadata workers.
- Normal ceiling: 160 GB total scraper ceiling including API, scheduler, adapters, processing workers, and caches.
- Emergency brake: if RSS exceeds 96 GB during live-search soak, stop browser workers first, set public search to queued-only, pause source-pack activation, and cut collection worker counts by half. If RSS approaches 160 GB, roll back scraper-native search and require explicit reallocation before continuing.
- 1 TB Inspur host reservation: keep scraper target at 96 GB and normal ceiling at 160 GB; reserve at least 500 GB RAM for API/frontend/search/DB/graph/model workloads and leave the remaining RAM as OS/page-cache/object-store headroom. Browser workers stay zero unless explicitly reallocated.
- Full CTI RAM reservation on a 1 TB host: scraper 96 GB target and 160 GB normal ceiling; API 16-32 GB; frontend 4-8 GB; Postgres 64-128 GB; OpenSearch/vector 128-192 GB; graph store 64-128 GB; object store/page cache 128-192 GB; OS cache plus emergency spare 160-256 GB. Keep at least 500 GB outside the scraper budget for the broader CTI app and model/search workloads before raising scraper workers.
- Browser crawling: budget separately at the process/container level. Do not borrow from the clear-web pool without reducing other workers.
- RAM caches are for hot metadata only. Raw evidence, captures, and queues should move to disk/object storage before scale testing.
- Worker-pool sizing must account for clear-web, processing, Telegram, browser, and darknet metadata pools separately. Keep browser workers at zero by default.
- Darknet metadata workers must remain low-concurrency; counts above 8 are treated as a configuration error.
- Capacity estimates include API baseline, per-worker memory, queue memory, and a safety margin. If the estimate exceeds the 96 GB target, reduce browser, clear-web, or processing workers before considering more RAM.

## Live Search SLOs
- Initial `/ti` response target: 1 second.
- First partial-result target: 5 seconds.
- Normal polling interval: 2 seconds; degraded polling interval: 10 seconds.
- Active-run limit: one active run per tenant/query. Reuse existing runs instead of creating duplicate live-search work.
- Provider failure budget: 2% of live-search requests.
- Zero-result budget: 25% of live-search requests before source activation gaps need review.
- Public `/ti` backpressure should reject or delay new runs when a tenant/query is already searching, the scraper/API proxy is degraded, the queue is backed up, no approved sources are available, or a darknet metadata kill switch is active.
- Live-search metrics should include provider failures, zero-result searches, duplicate active runs, polling volume, source activation gaps, and external dependency latency. Metric labels must stay low-cardinality and avoid raw query strings.
- Additional public-path metrics: fallback provider health, scraper-native `/v1/intel/search` health, temporary outer-wrapper fallback usage, and source-activation gaps.
- Polling impact at 2 seconds: 1 client is 0.5 req/s and about 65 MB ops overhead; 10 clients are 5 req/s and about 69 MB; 100 clients are 50 req/s and about 119 MB; 1,000 clients are 500 req/s and about 505 MB. CPU/API request pressure, not RAM, is the limiting factor at high polling counts.
- Slow polling when duplicate active runs appear, queue backlog grows, or provider latency exceeds the partial-result SLO. Disable external live discovery when the fallback provider is down, starts dominating responses, or consumes the failure budget. Return queued-only when the scraper is healthy but providers are degraded. Require login/API key when anonymous clients create sustained high polling volume, source activation probing, or repeated zero-result fanout.
- The outer fallback may be removed only when the scraper-native readiness gate passes: scraper endpoint healthy, random actor query returns partial evidence, run id present, cursor polling works, source coverage returned, degraded responses remain controlled, and public `/ti` proof still shows live partials.
- Roll back to the outer fallback if scraper-native `/v1/intel/search` returns non-2xx, omits run ids, breaks cursor polling, loses source coverage explanations, reports uncontrolled degraded/error states, or causes public `/ti` to stop showing live partial markers.

## Incident Runbooks
- Public canary collection: `POST /v1/sources/canary-activation` requires `operatorApproval=true` and activates only the bundled safe public clear-web portfolio. `POST /v1/ops/canary/run` and the background canary loop do not implicitly activate sources; they collect only already-active approved canary sources, create an auditable collection plan/run, queue bounded canary tasks, externalize public raw bodies through the object evidence boundary when configured, and report inserted/duplicate captures, retries, dead letters, freshness, error rate, duplicate rate, and promotion yield through `GET /v1/ops/canary`.

- Source breakage: move the source to `degraded` or `paused`, preserve last successful capture metadata, and open parser fixture work.
- Unsafe collection attempt: leave the source disabled or review-required, preserve the policy decision/audit event, and do not retry automatically.

## Main-Agent Alert Response
- Memory alert: fetch `/v1/ops/resource-snapshot`, compare current RSS to target, inspect worker pool sizing, set browser workers to zero if any are active, then reduce clear-web and processing concurrency until capacity is back under target. Do not raise above 160 GB without explicit reallocation.
- Queue age alert: inspect queued and leased counts, source health, retry/backoff state, and downstream storage latency. Add workers only if the capacity estimate remains below the 96 GB target and source-level limits are not the bottleneck.
- Adapter failure-rate alert: identify adapter and source IDs from structured logs, pause or degrade repeatedly failing sources, preserve warnings/last-success metadata, and create fixture coverage before re-enabling.
- Telegram rate-limit alert: inspect `scraper_adapter_rate_limit_delay_seconds`, delay the source window until `rateLimitResetAt`, and avoid increasing Telegram concurrency while the alert is active.
- Live search degraded: check `scraper_live_search_provider_failures`, `scraper_live_search_external_dependency_latency_ms`, and `/v1/ops/resource-snapshot`; then switch clients to the degraded poll interval and cached partials until provider failures clear.
- Live search wrapper fallback: check `scraper_live_search_outer_fallback_used`, fallback-provider health, and scraper-native endpoint health. Prefer scraper-native `/v1/intel/search`; keep the outer fallback temporary and alert while it is serving production traffic.
- Scraper-native cutover failure: restore the previous `api/src/utils/ti/search.ts` fallback path, redeploy `hanasand_api`, run `bun run check:live-search-deploy`, then keep `scraper_live_search_scraper_native_healthy=0` alerting until `bun run check:scraper-native-search` passes again.
- Cursor polling broken: stop new scraper-native cutover attempts, return queued-only or cached partial responses, and verify cursor fields plus delta payloads with Agent 09 before reenabling public polling.
- Source activation gap too high: keep public search in partial/blocked mode, surface activation recommendations, and ask Agent 01/02 to review source coverage before increasing fanout.
- 24h soak rollback: set the API wrapper back to the last known-good fallback, redeploy `hanasand_api`, verify `curl -fsS https://api.hanasand.com/api/ti/search?q=APT29`, run `bun run check:live-search-deploy`, then post the soak status template with rollback reasons to the leader thread and `coordination.md`.
- 24h soak escalation: page the main agent when RSS exceeds 96 GB, queue p95 exceeds 120s, worker saturation exceeds 95%, scraper-native health fails twice in a row, public proof fails, unsafe policy retries appear, stale-cache rate exceeds 10%, source-unavailable rate exceeds 20%, or duplicate active runs reappear after Agent 02 backpressure.
- Public-channel polling kill switch: disable public-channel source-pack activation, keep existing approved captures read-only, raise poll interval to 10 seconds, and require login/API key if anonymous public polling keeps pressure high.
- Restricted metadata kill switch: set restricted sources to `disabled` or keep `SCRAPER_DARKNET_METADATA_ONLY=true` with no active metadata workers; never retry policy-blocked restricted targets.
- Source-pack activation rollback: revert newly activated source packs to `candidate` or `needs_review`, keep provenance and activation dry-run logs, and rerun source activation coverage before restoring.
- Live search no approved sources: surface source activation recommendations, keep the run in `blocked` or `partial`, and do not bypass governance for Telegram, darknet, forums, or leak-site material.
- Live search queue backlog: reuse active runs, stop creating duplicate tenant/query runs, raise polling interval to 10 seconds, and reduce fanout before adding workers.
- Live search darknet kill switch: keep darknet metadata tasks disabled, return clear public-source partials only, and record the blocked state as an operational event rather than a retryable failure.
- Policy-block alert: verify the policy decision, keep high-risk or restricted sources review-gated, and do not convert blocks into automatic retries.
- Docker context drift alert: inspect `.dockerignore` first, then remove generated build artifacts from the context. Do not ship contexts containing `node_modules`, `.next`, coverage, or large unrelated directories.
