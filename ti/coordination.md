# TI Coordination

## Scraper Status
- Owner: main scraper agent.
- Runtime decision: Bun + TypeScript. Do not introduce Python for scraper core services.
- Current phase: foundation scaffold and multi-agent work split.
- Scope: enterprise CTI collection service with source registry, focused frontier, safe raw evidence handling, processing pipeline, API surface, and later Tor/I2P/Freenet metadata-only darknet adapters.
- Safety line: no authentication bypass, no CAPTCHA solving, no interaction with threat actors, no stolen-data downloads, no redistribution of leaked data. Darknet/leak sources store metadata, hashes, provenance, and optional screenshots only when explicitly approved.

## Architecture Baseline
- Source registry stores source type, access method, trust/reliability, language, cadence, legal notes, and last-seen state.
- Collection adapters emit a normalized `CollectedItem`.
- Focused frontier scores candidates using link context, parent relevance, source reputation, novelty, and safety policy.
- Raw evidence is immutable and separated from extracted intelligence.
- Processing pipeline normalizes, deduplicates, extracts IOCs/entities, maps ATT&CK/TTP candidates, and emits review-ready incident candidates.
- API exposes health, sources, queue status, captures, and pipeline results first; richer CTI endpoints come later.

## RAM Budget Draft For Inspur
- Total server RAM: 1 TB.
- Initial scraper reservation: 96-160 GB hard ceiling for production worker pool, with a default target below 96 GB.
- Suggested allocation at maturity:
  - Collection workers: 16-48 GB depending on dynamic browser concurrency.
  - Playwright pool: 32-96 GB only when dynamic crawling is enabled.
  - Text extraction and enrichment: 16-64 GB.
  - Queue/frontier/index sidecars: 8-32 GB.
  - Local caches: bounded by disk-first storage; RAM cache should be capped.
- Keep at least 500 GB free for the full CTI application, databases, model workloads, and operational headroom unless the leader explicitly reallocates.
