# Hanasand Share Page User Stories 1021-1040: Browser Proof In The Website UI

These stories test whether the `/s` share page can act like a real autonomous website builder rather than a terminal transcript. The user should see browser proof, pending file changes, and manual apply controls in the website UI.

## 1021. Live Or Imagined
As a founder, I ask "does this look live or am i kidding myself." I need browser proof before the agent claims the page works.

Success criteria:
- Browser proof appears in the share chat.
- The proof includes title, headings, links, and page errors.
- File changes remain pending until I apply them.

## 1022. Designer First Screen Review
As a designer, I say "first screen feels wrong, look at it." I need visible headings and CTA evidence before another rewrite.

Success criteria:
- Browser evidence is visible in `/s`.
- The agent uses evidence before proposing edits.
- The answer stays brief.

## 1023. Newbie Contact Path
As a total beginner, I ask where contact is. I need the UI to show links/buttons/forms, not raw tool JSON.

Success criteria:
- Contact/support links are visible in a browser proof card.
- Raw `hanasand-tool` tags are hidden.
- The next edit is scoped.

## 1024. Corporate Pricing Proof
As a corporate buyer, I need pricing to be obvious without fake prices.

Success criteria:
- Pricing links/headings are checked.
- Missing pricing becomes a focused edit.
- No unsupported pricing claim is made.

## 1025. Mobile Proof
As a mobile-first user, I say "phone users complain, dont just lint."

Success criteria:
- Viewport metadata is visible.
- Screenshot availability is honest.
- Lint alone is not treated as mobile proof.

## 1026. Agency Proof Before Apply
As an agency PM, I need proof before anything lands.

Success criteria:
- Browser proof appears before manual apply.
- Pending file changes are listed.
- Apply remains explicit.

## 1027. Public Sector Accessibility Basics
As a public-sector operator, I need screen-reader basics checked first.

Success criteria:
- Inputs/forms and headings are visible.
- Missing labels are concrete follow-up work.
- No broad compliance guarantee is made.

## 1028. Scammy Claims
As a founder, I say the page feels scammy.

Success criteria:
- Visible claims are inspected through browser proof.
- Risky promises are removed in pending edits.
- Caveats stay visible.

## 1029. Restaurant Booking Without Fake Booking
As a restaurant owner, I need booking cues without pretending a backend exists.

Success criteria:
- Buttons/forms are checked.
- Booking limitations are named.
- No fake checkout or reservation backend is claimed.

## 1030. Ops Visible Change Proof
As ops, I need to know what changed and whether users can see it.

Success criteria:
- Pending files are visible.
- Browser proof is visible.
- Summary is compact enough for handoff.

## 1031. Investor Link Today
As a founder, I am sending the link today.

Success criteria:
- Browser surface is checked before readiness claims.
- Biggest visible risk is named.
- Pending changes remain reviewable.

## 1032. Compliance Docs
As compliance, I need docs visible without an upload portal.

Success criteria:
- Docs/policy links are checked.
- Upload or portal claims are avoided.
- Missing proof becomes a next edit.

## 1033. Blank Page Complaint
As support, I say users see a blank page.

Success criteria:
- Page errors and headings are shown.
- The UI avoids reassurance without evidence.
- Screenshot limitation is explicit.

## 1034. Beginner Next Step
As a beginner, I need the next step to be obvious.

Success criteria:
- Buttons/forms are visible in evidence.
- The next edit sharpens only the next step.
- No long explanation is required.

## 1035. Generic Copy
As a designer, I say copy is generic.

Success criteria:
- Headings are inspected.
- The rewrite targets visible copy.
- Token use stays low.

## 1036. Support Page Friction
As support lead, I need buttons and forms checked before a support rewrite.

Success criteria:
- Buttons/forms appear in proof.
- The agent does not invent a ticket system.
- The pending diff stays small.

## 1037. Sales Overpromised
As sales ops, I need visible proof and caveats.

Success criteria:
- Browser proof anchors the claims review.
- Unsupported claims are removed.
- The summary names what is still not proven.

## 1038. Handoff To Another Agent
As a teammate, I need evidence persisted for the next agent.

Success criteria:
- Browser proof remains in the share chat.
- Pending files remain visible.
- The handoff does not require rereading terminal scrollback.

## 1039. Terminal-Agent Contrast
As a Codex or Claude terminal user, I need to see why the website UI is better.

Success criteria:
- Proof, pending changes, and apply state are visible.
- The answer acknowledges context/rate-limit pain without overclaiming.
- UI state is not hidden in logs.

## 1040. Drift Check
As product owner, I ask whether we are still building the best autonomous website builder.

Success criteria:
- The answer ties the change to faster project delivery.
- Evidence and apply controls are treated as product value.
- Test count is not mistaken for the goal.
